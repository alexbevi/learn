Agent2Agent Protocol and the Future of Agentic Workloads

A practical deep dive into why A2A exists, what changes when agents can collaborate across frameworks, and how production workloads need to adapt around discovery, tasks, state, security, and operations.

What you should be able to reason about

Protocol

The Friction

The hard part is not HTTP; it is agent semantics

Discovery What can this remote agent do, what modes does it accept, and how should I authenticate?
State How do I continue a task after clarification, approval, or a network interruption?
Progress How does a remote agent report working, blocked, completed, failed, or input-required states?
Artifacts Where do final deliverables live, and how are partial files or structured results streamed?
Opacity How can agents collaborate without exposing private tools, prompts, memory, or proprietary logic?
Operations How do auth, audit, retries, cancellation, tracing, and version negotiation survive composition?

Agents Are Not Tools

Tool wrappers flatten agent behavior into one call

Capability Tool-shaped wrapper A2A-shaped remote agent
Interaction One request, one structured response. Multi-turn task with clarification, state, progress, and artifacts.
Ownership Caller owns orchestration details and retry logic. Remote agent owns its internal workflow behind a stable contract.
Evolution API changes leak into every caller. Agent Card and protocol versioning create a negotiation boundary.

What A2A Is

A2A standardizes how agents expose and consume delegated work

Discovery contract

An Agent Card describes identity, interfaces, capabilities, security, modalities, and skills.

Work contract

Messages initiate or continue Tasks. Tasks produce status updates and Artifacts.

Transport contract

The 1.0 spec defines shared semantics across JSON-RPC, gRPC, and HTTP+JSON bindings.

The important shift: callers integrate with an agent surface, not a vendor-specific inner loop.

Protocol Shape

The layers separate data model, operations, and binding

Agent CardIdentity, skills, interfaces, security, capabilities.
Message + PartText, files, URLs, and structured JSON.
Task + ArtifactLifecycle state and concrete deliverables.

Discovery

Discovery strategy is an architecture decision

1 Known domain Fetch /.well-known/agent-card.json from a public or partner domain.
2 Private registry Search governed cards by skill, owner, environment, policy, or trust tier.
3 Direct config Use pinned card URLs for tightly coupled internal workflows and local development.
4 Authenticate Read declared schemes and obtain the right token or credential outside the message body.
5 Select interface Choose binding, version, modes, and tenant before sending work.

Lifecycle

A request becomes a task, then events and artifacts

1DiscoverClient gets Agent Card and chooses a skill/interface.
2Send messageClient sends a Message with one or more Parts.
3Create taskAgent returns a Task or direct Message.
4Track workPolling, streaming, or push delivers status and artifact updates.
5ContinueClient sends more input using contextId, taskId, or references.

A minimal portable call is just protocol-shaped HTTP

Task States

Workloads need to handle interrupted states, not only success/failure

In flight

TASK_STATE_SUBMITTED and TASK_STATE_WORKING mean the agent accepted work and may emit progress.

Interrupted

TASK_STATE_INPUT_REQUIRED and TASK_STATE_AUTH_REQUIRED mean the caller must continue the interaction.

Terminal

COMPLETED, FAILED, CANCELED, and REJECTED close the task's execution path.

Artifacts

Messages are conversation; artifacts are deliverables

Message

Status text, clarification, progress update, or instruction. Useful for user-facing narration and task continuation.

Decision record, report, invoice draft, JSON payload, file reference, generated media, or other concrete output.

Security

A2A delegates auth to standard web security, but composition raises the bar

Card authenticityUse HTTPS, signed cards where appropriate, and governed registries for private estates.
Caller identitySecurity schemes are declared in the card; credentials travel in headers, not prompt text.
Authorization scopeAgents must restrict task visibility and artifacts to authorized clients.
Webhook safetyPush requires SSRF protection, origin verification, replay handling, and idempotent receivers.
Prompt boundaryRemote agent messages are untrusted inputs; validate artifacts before execution or persistence.
AuditRecord who delegated what to which agent version under which card and policy.

A2A And MCP

Use MCP for tools; use A2A for peer agents

MCP side

Agent uses capabilities Database lookup, search, calendar, file system, code runner, product API.

Usually structured, tool-like, and owned by the calling agent.

+

A2A side

Agent partners with agents Refund specialist, travel planner, procurement agent, legal reviewer, operations analyst.

Usually stateful, multi-turn, policy-bearing, and internally opaque.

The system becomes a network of capability-owning agents

What Changes

The “after” is not magic; it moves responsibilities

From integration code to agent contracts

Teams publish capabilities and lifecycle semantics instead of private SDK assumptions.

From monolith agent to agent mesh

Coordinators select remote specialists at runtime based on cards, trust, cost, latency, and policy.

From function output to task state

Systems need durable task stores, event processing, idempotency, cancellation, and artifact validation.

Workload Architecture

A2A fits best at service boundaries, not inside every loop

Inside a serviceFramework-native graph, tools, prompts, model calls, local memory.
Use direct callsFast internal functions and deterministic tools do not need A2A overhead.

A2A boundary

Expose the useful delegation surface Stable skills, accepted modes, task lifecycle, artifacts, auth, and observable events.
Across servicesDifferent teams, languages, frameworks, vendors, trust domains, or release cycles.
Use A2ALong-running, interactive, stateful, or opaque specialist work.

Adoption Path

Practical adoption is a wrapping and governance exercise

1. Inventory agents Identify which workflows are true remote specialists and which are just tools or APIs. Do not wrap everything
2. Define cards Write skills, modes, examples, security, supported interfaces, and operational owners. Contract first
3. Add adapters Bridge framework state to A2A messages, task store, artifacts, streaming, and cancellation. Framework-specific work
4. Govern runtime Registry, auth, trace correlation, retries, cost limits, rate limits, and safety gates. Production platform

LangGraph Example

Expose message-based graph state as an A2A-compatible agent

from typing import Any, TypedDict
from langgraph.graph import StateGraph

class State(TypedDict):
    messages: list[dict[str, Any]]

async def call_model(state: State) -> dict[str, Any]:
    last = state["messages"][-1]["content"]
    decision = await refund_chain.ainvoke({"request": last})
    return {"messages": state["messages"] + [{
        "role": "assistant",
        "content": decision.summary
    }]}

graph = StateGraph(State)
graph.add_node("refund_review", call_model)
graph.set_entry_point("refund_review")
graph.set_finish_point("refund_review")
agent = graph.compile()

# Deployed through an Agent Server A2A endpoint.
# Client A2A contextId maps naturally to a conversation thread.

Adapter Work

Every framework still has to bridge the same concepts

State mapping

Map contextId, taskId, message history, and continuation into the framework's session or graph state.

Task persistence

Keep task status, history, artifacts, callback configs, and cancellation state outside ephemeral model calls.

Streaming bridge

Convert framework events into A2A status and artifact updates without leaking internal chain-of-thought or private traces.

Client Code

A portable client has five responsibilities

import httpx

async def delegate(card_url: str, token: str, text: str) -> dict:
    async with httpx.AsyncClient(timeout=30) as http:
        card = (await http.get(card_url)).json()
        iface = next(
            i for i in card["supportedInterfaces"]
            if i["protocolBinding"] == "HTTP+JSON" and i["protocolVersion"] == "1.0"
        )
        response = await http.post(
            f'{iface["url"]}/message:send',
            headers={
                "Authorization": f"Bearer {token}",
                "A2A-Version": "1.0",
                "Content-Type": "application/a2a+json",
            },
            json={
                "message": {
                    "messageId": "msg-1842",
                    "role": "ROLE_USER",
                    "parts": [{"text": text, "mediaType": "text/plain"}],
                },
                "configuration": {"returnImmediately": True},
            },
        )
        response.raise_for_status()
        return response.json()

Routing

The coordinator becomes a runtime router, not a giant prompt

def choose_agent(goal: str, cards: list[dict]) -> dict:
    candidates = []
    for card in cards:
        for skill in card["skills"]:
            if semantic_match(goal, skill["description"], skill.get("tags", [])):
                candidates.append((card, skill))

    return rank(
        candidates,
        by=["required_auth_available", "trust_tier", "latency_slo", "cost_budget"],
    )[0][0]

async def handle_customer_request(request: str):
    card = choose_agent(request, registry.list_cards())
    task = await delegate(card["cardUrl"], token_for(card), request)
    return await wait_for_result(task)

State And Identity

Long-running delegation needs durable correlation

contextId

Conversation or collaboration continuity across multiple tasks.

taskId

Specific unit of work with lifecycle, history, status, and artifacts.

Trace ID

Observability correlation across coordinator and remote agent boundaries.

Principal

User, service account, tenant, scopes, consent, and delegated authority.

Data Contracts

A2A transports structured data, but it does not design your domain schema

Use Part.data

For machine-readable order facts, policy inputs, entity IDs, and decision records.

Validate artifacts

Use JSON Schema, Pydantic, Zod, or internal contracts before downstream automation.

Version domain payloads

A2A protocol version is not a substitute for your business schema version.

Long-Running Work

A2A makes human-in-the-loop and slow operations first-class

Submit task Coordinator asks a refund specialist to evaluate a high-value refund. Task: submitted
Require input Specialist requests manager approval or asks for missing shipment evidence. Task: input-required
Resume Coordinator sends approval back with the same task or context identifiers. Task: working
Complete Specialist returns a decision artifact and a user-facing explanation. Task: completed

Failure Modes

Agent networks fail differently than single-agent apps

Card driftCaller cached old skills or interface URLs after a remote deployment.
Semantic mismatchThe selected skill accepts the request but interprets domain terms differently.
Partial completionTask returns artifacts but misses required fields or final business approval.
Callback lossPush notification fails or duplicates; receiver must fetch task state idempotently.
Auth interruptionRemote agent needs a user-scoped token mid-task.
Prompt injection propagationRemote agent output enters another agent unless artifacts are validated and scoped.

Observability

Trace the protocol boundary, not only the model call

Protocol events

Card fetched, interface selected, message sent, task created, status changed, artifact emitted.

Business events

Refund approved, logistics exception detected, human approval requested, customer notified.

Operational events

Auth failure, version mismatch, timeout, retry, cancellation, callback delivery, rate limit.

Governance

A2A makes agent publishing look more like API product management

Ownership

Every card needs an owner, support channel, version policy, uptime target, and deprecation path.

Trust

Registries should encode trusted issuers, allowed callers, scopes, environments, and data classification.

Evaluation

Specialist agents need contract tests and task-level evals, not only chat transcript review.

Workload Evolution

To benefit from A2A, workloads need to become task-native

Durable task stores

Tasks, status, history, artifacts, callback config, and cancellation state need persistence.

Event-driven orchestration

Coordinators should react to status and artifact events rather than blocking every request thread.

Contracted outputs

Artifacts should be typed, validated, versioned, and suitable for downstream automation.

Anti-Patterns

A2A does not fix weak agent design

Wrapping everythingSimple deterministic APIs should stay tools or services.
Prompt-only contractsIf the output drives automation, require structured artifact validation.
Trusting remote textTreat remote messages as untrusted content until validated or reviewed.
No cancellationLong-running work needs cancel, timeout, retry, and compensation behavior.
No registryHardcoded card URLs do not scale across teams, environments, and deprecations.
Hidden handoffsUsers and operators need auditability when work crosses agent boundaries.

Maturity

Expect a mixed world for a while

Protocol is ahead of platforms

The 1.0 spec is stable enough to learn, but framework implementations vary by binding, version, and operational depth.

Adapters will matter

Teams will need thin wrappers for OpenAI Agents SDK, custom services, older LangChain apps, and domain-specific agents.

Enterprises will add control planes

Registry, identity, policy, observability, cost governance, and approvals become the real differentiators.

Concrete Example

Before and after for customer operations

Concern Before A2A After A2A
Refund specialist Called through a custom REST endpoint with bespoke payloads and no task state. Publishes a card and returns task updates plus a structured decision artifact.
Logistics specialist Wrapped as a tool, losing its ability to ask clarifying questions. Runs as a remote agent that can request more evidence and stream investigation progress.
Coordinator Hardcodes integrations and owns every retry, timeout, and domain nuance. Selects specialists from a registry and manages user experience, policy, and final response.

Recap

The mental model

Before

Agents are isolated apps or tool-shaped wrappers connected by brittle, custom glue.

A2A

A standard contract for discovering agents, delegating tasks, exchanging multimodal parts, and tracking artifacts.

After

Agentic workloads become distributed systems: task-native, event-driven, governed, observable, and framework-agnostic at the boundary.
