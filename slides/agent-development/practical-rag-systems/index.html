Practical RAG Systems

A technical product and solutions guide to retrieval-augmented generation: two production use cases, system diagrams, evaluation loops, and a database compare/contrast across MongoDB, pgvector, vector-first stores, and search platforms.

80 minutes / 43 slides / Last updated 2026-06-01

Build the architecture conversation

System flowTrace ingestion, retrieval, reranking, generation, citation, and feedback.
Use casesFollow support escalation and proposal/compliance workflows end to end.
Database fitCompare MongoDB, pgvector, vector-first stores, and search engines by workload boundary.
RolloutDefine evals, observability, risk controls, and adoption gates.

Practical promise

RAG is a trust product, not an embedding feature

Demo promise Upload files, ask natural-language questions, and get fluent answers over a sample corpus.
Production promise Answer with current, permissioned, source-cited evidence that fits a real operational workflow.

The durable value is governed access to external knowledge: customer context, product docs, tickets, policies, contracts, incidents, and source systems that change after model training.

Boundary

What RAG owns

FindIdentify sources, chunks, records, entities, and constraints relevant to a question.
FilterApply tenant, role, product, version, date, region, and approval constraints.
GroundPack evidence into a prompt so the model answers from cited context.
ImproveLog misses, feedback, source drift, and eval failures back into the system.

RAG does not make source truth disappear. It makes source truth queryable and usable by a model at runtime.

Reference architecture

A production RAG system has loops

Source systemsDocs, tickets, CRM, contracts, policies, incidents, product telemetry.
Source ledgerOwner, URL, version, effective date, permission scope, checksum.
Ingestion loopParse, normalize, chunk, embed, index, validate freshness.
Query loopClassify intent, rewrite, filter, retrieve, fuse, rerank, pack context.
Answer loopGenerate, cite, verify, refuse, route to human when needed.
Evaluation loopGold set, trace review, retrieval metrics, answer faithfulness, outcomes.
OperationsLatency, cost, index health, source drift, feedback, incident response.

Use case anchors

Two workflows that expose real constraints

Support escalation copilot High-severity issue triage over docs, incidents, resolved tickets, customer config, logs, and known bugs.
Proposal and compliance engine RFP, security questionnaire, and compliance answers over approved claims, controls, policies, and clauses.
Shared pressureExact identifiers, permissions, freshness, citations, evals, and human handoff.
Different riskSupport optimizes time-to-resolution; proposal workflows optimize authority and liability control.

Use case 1

Support escalation copilot

UserSupport engineer handling a high-severity customer incident.
Question"Customer on version 7.1 sees E11000 after region failover. Known issue?"
Required contextProduct version, customer config, incidents, KB, tickets, release notes, bug status.

A generic semantic-search demo fails here because the important clues are mixed: exact error codes, versions, customer-specific state, and natural-language symptoms.

Evidence model

Metadata is product logic

SourceKB, docs, incident, ticket, bug, runbook, customer config.
ScopeTenant, account, role, visibility, support tier, disclosure rule.
ProductComponent, version, cloud region, feature flag, deployment type.
FreshnessEffective date, fix version, superseded-by, last verified, owner.

The database needs these fields because retrieval, authorization, citations, and debugging all depend on them.

Ingestion flow

Do not make the vector index the source ledger

AcquireConnectors pull docs, tickets, incidents, bugs, and config snapshots.
NormalizeParse, clean, classify, and attach owner, version, permission, and product metadata.
ChunkSplit by symptom, cause, workaround, command, fix, caveat, and source boundary.
Embed + indexStore vectors, full text, exact fields, filters, and source pointers.
ValidateCheck freshness, empty chunks, permission scope, and index availability.

Chunking

Chunk around problem-solving units

SymptomObserved behavior, error message, log signature, customer wording.
DiagnosisLikely cause, affected versions, preconditions, excluded causes.
ActionVerification command, workaround, rollback, fix, escalation path.
CaveatRisk, support tier, region, version exception, deprecation, supersession.

Good chunks carry enough local context to be useful after retrieval. Tiny fragments lose meaning; giant chunks drown the answer prompt.

Query planning

Preserve exact clues before semantic retrieval

ParseExtract product, version, error code, tenant, region, date, urgency.
AuthorizeResolve account, role, visibility, and source access.
RewriteAdd synonyms and diagnostics without dropping original identifiers.
RouteChoose docs, tickets, incidents, bugs, config, or all-of-the-above retrieval.
PlanSet filters, candidate counts, reranking, and citation requirements.

Support retrieval

Hybrid retrieval mirrors the real question

FiltersTenant, role, product, version, date, region, visibility.
LexicalError codes, command names, SKUs, flags, exact phrases.
VectorSymptoms, paraphrases, concept matches, related incidents.
RerankScore query-document pairs for top-k precision.
PackDiverse, authoritative, fresh evidence with citations.

Answer contract

The output shape is part of the product

{
  "diagnosis": "Likely duplicate-key behavior after regional failover.",
  "evidence": ["KB-1842", "INC-2026-0417", "BUG-91833"],
  "next_checks": ["Confirm version", "Check failover timestamp", "Inspect unique index state"],
  "recommended_action": "Apply workaround only if the customer matches the affected version range.",
  "confidence": "medium",
  "escalate_if": ["data loss suspected", "unsupported version", "evidence conflicts"]
}

The schema gives support engineers, auditors, and eval jobs the same fields to inspect.

Operations loop

Every bad answer should produce a fixable artifact

MissRelevant source was absent, stale, hidden, or badly chunked.
DiagnoseInspect trace: query, filters, candidates, scores, rerank, prompt.
FixUpdate source, metadata, chunking, index, prompt, or answer policy.
BenchmarkAdd or update gold query, expected evidence, and acceptance metric.
RedeployMeasure quality, latency, cost, and user outcome after change.

Support decision

Choose the platform by the hardest boundary

MongoDBStrong if tickets, docs, customer config, chunks, metadata, and workflow state are document-shaped or already MongoDB-backed.
pgvectorStrong if support data lives in Postgres and SQL joins, ACID, backups, and governance are the central boundary.
Vector-firstStrong if vector serving scale, ANN operations, multi-vector retrieval, or managed vector performance dominates.
Search engineStrong if exact tokens, analyzers, synonyms, enterprise search operations, and hybrid rank pipelines dominate.

Use case 2

Proposal, security, and compliance answer engine

UserSolution architect, security reviewer, seller, legal reviewer, or partner engineer.
Question"Do we support customer-managed keys in EU regions for regulated workloads?"
Required contextApproved answer library, policies, SOC reports, product docs, roadmap rules, clauses.

The risk is not just a wrong answer. It is an unauthorized answer, an outdated answer, or an answer that cites the wrong authority.

Authority model

Not every relevant source is usable

DraftUseful evidence, not approved for customer-facing use.
InternalReviewer guidance only; do not quote externally.
ApprovedSafe for specified segments, regions, products, and dates.
ExceptionCustomer-specific answer with explicit owner and expiry.
ExpiredRetained for audit, blocked from auto-answer retrieval.

Proposal ingestion

Chunk by evidence unit, not page length

Controlled sourcesPolicies, controls, clauses, public docs, answer library, SOC reports.
Evidence unitsQuestion-answer pair, control, clause, policy section, exception note.
Authority metadataOwner, approval state, disclosure level, effective date, expiry.
IndexesText, vector, filters, product tags, compliance framework tags.
Audit trailSource version, reviewer, answer reuse, customer-specific exceptions.

Question classification

Route before retrieval

Product factRetrieve approved docs and answer library entries.
Security controlRetrieve mapped controls, evidence, reports, and reviewer notes.
Contract clauseRetrieve legal templates, fallbacks, and customer exceptions.
Roadmap claimRoute through explicit policy; often requires human review.
Ambiguous askAsk a clarification or draft with lower confidence.
ConflictSurface disagreement and route to owner instead of choosing silently.

Governed retrieval

Block unusable evidence before the prompt

IdentityUser role, customer, partner, region, opportunity, disclosure level.
Policy filtersApproved state, effective date, customer segment, product, geography.
SearchHybrid retrieval over approved answer units and supporting sources.
Review gateRequire human approval for low confidence, conflict, exception, or roadmap.
AnswerDraft with citations, owner, confidence, and review state.

Review workflow

Human approval is part of the architecture

DraftModel proposes answer from approved evidence.
CheckSystem validates citations, conflicts, freshness, and policy.
ReviewOwner handles sensitive, stale, conflicting, or low-confidence answers.
ApproveAnswer becomes reusable for matching scope.
ExpireReview date or source drift removes auto-use.

Lifecycle

The workflow creates new governed knowledge

QuestionCustomer asks through RFP, security portal, email, or seller workflow.
DraftRAG answer cites source authority and flags risk.
ApproveHuman owner edits, accepts, limits, or rejects.
ReuseApproved answer becomes retrieval evidence with scope and expiry.
AuditTrack source, reviewer, customer, version, and downstream usage.

Retrieval patterns

Match the query shape

Dense vectorParaphrase, concepts, symptoms, natural-language similarity.
LexicalError codes, product names, legal terms, commands, identifiers.
StructuredTenant, role, region, date, product, version, approval state.
HybridDense plus lexical candidates merged by score or rank fusion.
Graph or relationalDependencies, customers, entitlements, controls, product relationships.
RerankedSecond-stage relevance over candidate chunks before context packing.

Signals

Dense, lexical, and structured each catch different truth

Signal Best at Weak at Production example
Dense vector Meaning, paraphrase, similar symptoms Exact short tokens and hard constraints "Failover duplicate writes" matching an incident summary
Lexical Error codes, names, clauses, commands Synonyms and implied intent E11000, CMK, "data residency"
Structured Permissions, tenant, region, date, version Open-ended semantic relevance EU-only, product 7.1, approved external disclosure

Hybrid search

Fusion is where search becomes a system

Lexical candidatesBM25, phrase, analyzer, exact-token or sparse-vector signals.
Vector candidatesDense similarity over semantically related chunks.
MergeRank fusion or score normalization with system-specific tuning.
DeduplicateCollapse sibling chunks, source variants, and repeated answers.
RerankOptional query-document scoring for precision.
PackContext chosen by authority, diversity, recency, and token budget.

Do not compare raw scores across systems without understanding how each system normalizes, fuses, and reports relevance.

Filters

Permissions are retrieval constraints

SecurityTenant, role, source visibility, disclosure class, customer scope.
BusinessProduct, tier, region, industry, customer segment, opportunity type.
TimeEffective date, expiry, incident window, version range, last verified.
RiskApproval state, legal sensitivity, roadmap claim, exception-only evidence.

Filtering after retrieval can expose the wrong evidence to ranking logic and reduce recall. The safer default is to constrain candidate generation as early as the store supports.

Reranking

Use it when candidates are good but ordering is noisy

Retrieve broadCollect 25-200 candidates from hybrid or multi-query retrieval.
Score pairsReranker evaluates query and candidate text together.
SortPromote precise evidence and demote near misses.
TrimKeep top evidence that fits context budget and source diversity.
MeasureWatch added latency, cost, and whether answer quality improves.

Context packing

The prompt should receive evidence, not a junk drawer

RelevanceEvidence directly addresses the specific question.
AuthoritySource has correct owner, approval, and disclosure state.
DiversityAvoid ten chunks that say the same thing.
FreshnessPrefer current source versions and expose conflicts.

Context packing is where retrieval becomes answer policy: what gets included, excluded, summarized, quoted, cited, or escalated.

Failure modes

Debug by pipeline stage

Symptom Likely stage Fix to test
Relevant source never appears Ingestion, chunking, metadata, filters, index freshness Inspect source ledger, chunk text, filter fields, and index status
Exact code or clause is missed Lexical retrieval or query planning Add keyword path, preserve tokens, tune analyzers, add fielded search
Right evidence, wrong answer Context packing, prompt, model, answer policy Change answer contract, citation rules, refusal checks, or model
Answer leaks internal source Authorization and pre-filtering Move policy checks before retrieval and add trace assertions

Source truthRaw documents, canonical records, source versions, provenance.
Retrieval indexVectors, text indexes, filters, chunks, scores, candidate ids.
Workflow stateJobs, approvals, drafts, reviews, escalation, user feedback.

MongoDB angle

Document-shaped RAG applications

Good fitEvidence, chunks, metadata, source lineage, workflow state, and product records are naturally documents.
Retrieval surfaceAtlas Vector Search for embeddings, Atlas Search for text, aggregation for app-side composition.
Why it mattersOne operational model can hold source records, metadata filters, vectors, traces, and answer workflows.
WatchIndex/search-node sizing, query stage constraints, filtered query cost, index freshness, and benchmark evidence.

pgvector angle

Vector search inside Postgres

Good fitPostgres already owns the application records, access model, reporting, and transactional workflow.
Retrieval surfaceExact and approximate nearest-neighbor search with vector types, distance metrics, HNSW, and IVFFlat.
Why it mattersSQL, joins, ACID, backups, row-level controls, and team familiarity stay in the center.
WatchANN tuning, index memory, recall/latency tradeoffs, operational isolation, and workload contention.

Vector-first systems

When retrieval serving deserves its own tier

PineconeManaged vector/search service with semantic, full-text, sparse, hybrid, metadata filtering, and production knobs.
WeaviateObject plus vector model with hybrid search, filters, named vectors, and integrated vectorization patterns.
QdrantPayload metadata, payload indexes, filtering, and vector search engine focus.
MilvusLarge-scale vector database with ANN and multi-vector hybrid retrieval patterns.
Architecture costYou still need source truth, permissions, workflow state, and evaluation somewhere.
Best pressureVector scale, specialized retrieval ops, multi-vector search, or independent retrieval service ownership.

Search engines

When lexical search and rank pipelines dominate

OpenSearchHybrid search pipelines combine keyword and semantic results with normalization or rank processors.
Azure AI SearchHybrid query and semantic ranker patterns for enterprise search plus vector retrieval.
Elasticsearch familySearch-engine lineage: analyzers, synonyms, BM25, vector fields, and operational search tooling.
Strongest fitExisting enterprise search tier, exact matching, analyzers, document search operations, and hybrid ranking.
WatchSource-of-truth split, permissions, index sync, application state, and answer workflow ownership.
Product lensOften best when users already expect a search experience upgraded with generated answers.

Local stores

Chroma and the prototype-to-production jump

Useful forLocal demos, embedded workflows, small self-hosted systems, eval harnesses, and fast iteration.
Retrieval surfaceEmbeddings, metadata, dense and sparse search, filtering, and full-text or regex search patterns.
Production questionWhere do source truth, authorization, backups, approvals, traces, and regression history live?
Good pathUse local stores to learn corpus behavior, then choose the durable operating boundary deliberately.

Architecture choices

Three common data-plane patterns

One-store RAGApplication data, chunks, vectors, filters, traces, and workflow state live together. Simpler ownership; watch retrieval scale.
App DB + search tierApp database owns truth and workflow; search or vector tier owns retrieval serving. Strong separation; requires sync and policy alignment.
Search-first RAGEnterprise search platform owns discoverability and hybrid relevance; application layer owns answers, approvals, and telemetry.

Splitting systems is justified when the split gives clear scale, operations, or relevance value. Otherwise it adds governance and freshness work.

Evaluation

RAG quality is layered

Business outcomeResolution time, answer reuse, review time, deflection, win support.
Answer qualityCorrectness, faithfulness, citation quality, refusal behavior.
Context qualityRelevant, diverse, current, authorized, sufficient evidence.
Retrieval qualityRecall@k, MRR, nDCG, exact-token recall, filter correctness.

Rollout plan

Ship trust gradually

Scope corpusPick one workflow, source set, owner map, and permission model.
Build gold setReal questions, expected evidence, expected answer behavior.
Benchmark offlineCompare chunking, retrieval, database, rerank, and prompt variants.
Pilot with reviewHuman approval, trace review, feedback capture, failure taxonomy.
Expand with gatesMonitor quality, cost, latency, source drift, and user outcomes.

Recap

Practical RAG is governed retrieval plus product workflow

Start with workflowSupport, proposal, compliance, and field teams each need different answer contracts.
Model the evidenceSource ownership, metadata, permissions, versioning, and freshness are core data design.
Choose the data planeMongoDB, pgvector, vector-first, and search engines win under different boundaries.
Operate the loopTrace, evaluate, review, and feed failures back into source and retrieval improvements.
