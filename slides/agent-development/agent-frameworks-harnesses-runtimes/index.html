Agent Frameworks, Harnesses, and Runtimes

A practical map of the agentic development stack: what each layer does, where framework boundaries sit, and how databases become durable product infrastructure.

70 minutes / 42 slides

Learning goals

By the end, the stack should feel separable

Layer boundaries

Separate model APIs, UI SDKs, agent harnesses, orchestration runtimes, low-code builders, observability, and databases.

Tooling value

Explain what LangChain, Mastra, n8n, Agno, Langflow, Flowise, LlamaIndex, ADK, OpenAI Agents SDK, Vercel AI SDK, and Microsoft Agent Framework actually add.

Database fit

Know when MongoDB, Postgres, pgvector, or another store is serving app state, retrieval, memory, checkpoints, audit, or eval data.

Design choices

Choose a stack from the shape of the work, not from a ranking of frameworks.

Running example

Support operations agent

Ticket

A customer says order A-123 is late, wants a refund, and threatens to churn. The agent must investigate, propose an action, and avoid unauthorized side effects.

Why this is useful

  • Needs structured tools, retrieval, memory, workflow state, approval, and traces.
  • Can be built code-first, low-code, or as a hybrid.
  • Exposes where databases add value beyond vector search.

Mental model

The stack is layered, but products blur layers

Experience Chat UI, ticket UI, internal tool, webhook, voice, Slack, email. Vercel AI SDK UI, n8n triggers, Langflow chat, custom app.
Harness Prompt, tools, model loop, structured output, middleware, guardrails. LangChain, Mastra, OpenAI Agents SDK, ADK, LlamaIndex, Agno.
Runtime State graph, durable execution, checkpoints, interrupts, streaming. LangGraph, Mastra workflows, Microsoft Agent Framework workflows, ADK workflows.
Data plane App records, policy docs, embeddings, memory, audit, traces, eval datasets. MongoDB, Postgres, pgvector, vector databases, object stores.
Operations Observability, evals, prompt management, deployment, access control. LangSmith, Langfuse, Mastra observability, Agno AgentOS, provider consoles.

Boundary language

Use precise names or the stack becomes muddy

Term Meaning in this deck Support agent example
Model API Generates text, tool calls, embeddings, or multimodal output. Classifies customer intent or drafts the response.
Harness Owns the model loop and decides how tools, prompts, context, and guardrails are applied. Lets the model call getOrder and searchPolicy.
Runtime Owns long-running state, checkpoints, deterministic routing, and resumption. Pauses before refund approval, resumes after manager review.
Control plane Manages agents, versions, traces, evals, credentials, schedules, and access. Shows failed refund attempts and regression scores by release.

First decision

Do not start by picking a framework

Use a function

When the steps and inputs are deterministic, write normal code. The model may only classify or summarize.

Use a harness

When the model should choose among tools, reason over context, and produce structured output.

Use a runtime

When state, approvals, retries, branches, parallel work, or resumption are business requirements.

The support agent uses all three: deterministic policy checks, a tool-using agent loop, and a workflow for high-risk actions.

Layer 1

Model and provider APIs are the substrate

What they solve

  • Text generation, tool-call generation, embeddings, vision, audio, and streaming.
  • Provider-specific model features, cost, latency, safety controls, and context windows.
  • Low-level request/response contracts.

What they do not solve

  • Business workflow state.
  • Reliable tool authorization.
  • Long-term memory semantics.
  • Evaluation and product ownership.

Vercel AI SDK

UI and application integration can be its own layer

Boundary

Vercel AI SDK is excellent when the product surface is a web app: typed tool definitions, streaming, model providers, UI helpers, and generative UI patterns.

Support agent use

Stream the draft response into the ticket UI, show tool status, and render an approval component when the agent proposes a refund. 

const result = await streamText({
  model,
  tools: { getOrder, searchPolicy },
  stopWhen: stepCountIs(6),
  prompt: ticket.message,
});

OpenAI Agents SDK

Provider-native agent runtime with few primitives

Agent

Instructions plus tools around a model, with an SDK-managed loop.

Handoffs

Delegate to specialized agents, such as a policy reviewer or fraud analyst.

Sessions and tracing

Persist working context and inspect tool calls, guardrails, and runs.

Use it when you want a lightweight Python-first path that stays close to OpenAI model behavior while adding tools, guardrails, sessions, handoffs, and tracing.

Layer 2

The harness is where "agent" becomes product behavior

Instructions Role, policy, tone, constraints, stop criteria.
Context Ticket, customer, policies, prior interactions.
Tools Typed actions with auth, schemas, logging, and side effects.
Loop Decide, call, observe, continue, stop, stream.
Output Draft, action plan, citations, risk flags, next step.

Tools

Tools are the hard boundary between language and action

const issueRefund = tool({
  description: "Create a refund request. Requires manager approval for high-value orders.",
  inputSchema: z.object({
    orderId: z.string(),
    amountCents: z.number().int().positive(),
    reasonCode: z.enum(["late_delivery", "damaged", "policy_exception"]),
  }),
  execute: async ({ orderId, amountCents, reasonCode }, context) => {
    await authorize(context.user, "refund:create");
    return refundService.createRequest({ orderId, amountCents, reasonCode });
  },
});

The framework can expose the tool, but your application owns authorization, idempotency, validation, audit, and compensating actions.

LangChain

LangChain is the broad harness and integration ecosystem

Value added

  • Agent loop around models, tools, prompts, middleware, structured output, and memory.
  • Large integration surface across model providers, retrievers, vector stores, and tooling.
  • Middleware for context management, guardrails, retries, human steering, and execution environments.

Boundary

LangChain is a strong default harness. When the support flow becomes a stateful business process, push orchestration into LangGraph rather than hiding every branch in prompt text.

Mastra

Mastra packages the TypeScript agent stack as one product shape

Agents

TypeScript agents with tools, structured output, processors, approvals, and multi-agent patterns.

Workflows

Step-based workflows with schemas, state, suspend/resume, streaming, and time travel.

Developer loop

Studio, server adapters, memory, RAG, observability, and evals are presented as framework-level concerns.

For a Node or Next.js product team, Mastra competes by reducing stack seams: application code, agent logic, and workflow code can live in TypeScript.

Google ADK

ADK is an agent app framework with workflow agents and runtime services

Primitive Boundary Support agent use
LlmAgent Reasoning worker using an LLM and tools. Drafts the customer response and calls policy tools.
Template workflow agents Deterministic sequential, loop, and parallel orchestration. Classify, retrieve, draft, approve, then update in a fixed path.
ADK 2.0 graph workflows Graph routes with data handling, human input, and dynamic workflow control. Branch support tickets by policy risk, missing data, or approval outcome.
Sessions, state, memory Short-term run context and long-term recall are separate concepts. Keep ticket state distinct from customer-level memory.

Microsoft Agent Framework

Microsoft's stack makes agents and workflows first-class

Agents

Individual agents process inputs, call tools and MCP servers, and generate responses across providers such as Microsoft Foundry, Azure OpenAI, OpenAI, Anthropic, and Ollama.

Workflows

Graph-based workflows connect agents and functions with type-safe routing, checkpointing, and human-in-the-loop support.

If you have heard "FAgent Framework," the public Microsoft Learn name is Microsoft Agent Framework.

LlamaIndex

LlamaIndex starts from the data problem

Where it shines

Loading, parsing, indexing, retrieving, and composing knowledge over private data, then exposing query engines and tools to an agent.

Support agent use

Build a policy and order-history retrieval layer first, then let the support agent call it with citations and metadata filters.

If your hardest problem is trustworthy retrieval over messy enterprise data, a data-first framework changes where you start.

Agno

Agno leans into the agent platform/control-plane problem

Build

Use the Agno SDK or bring agents built with another framework.

Run

Run agents as services with scheduling, tracing, RBAC, and audit trails.

Manage

Use AgentOS as a control plane while keeping data in your own infrastructure.

This is valuable when agent operations, governance, and ownership matter as much as the agent loop.

Layer 3

Runtimes exist because real work outlives one model call

State

What has happened, what is pending, what can be retried?

Control

Which branch runs, which steps are deterministic, which are model-driven?

Interrupts

Where must a human approve, edit, or reject?

Durability

Can the work resume after a crash, deploy, timeout, or queue delay?

LangGraph

LangGraph is the stateful runtime under many LangChain agents

Boundary

Model and tool behavior live in nodes. State, edges, checkpoints, interrupts, streaming, and recovery live in the graph runtime.

Support agent use

Route low-risk tickets to auto-response, high-risk tickets to approval, and fraud-risk tickets to a separate investigator node.

Graph path
classify
retrieve
draft
approve or execute

Mastra workflows

Mastra uses workflows for explicit TypeScript processes

Agent when open-ended

The model decides which support tools to call and how to synthesize the answer.

Workflow when ordered

The process defines classify, retrieve, draft, approval, refund request, notification, and audit write.

This is the core distinction: agents improvise within constraints; workflows encode the business process.

Runtime comparison

Google and Microsoft also draw the agent/workflow boundary

Framework Agent side Workflow side When it fits
Google ADK Agents use LLMs, tools, sessions, state, and memory. Workflow agents and ADK 2.0 graph workflows control execution. Google/Gemini-heavy stacks, multi-agent app patterns, and deployable agent services.
Microsoft Agent Framework Agents call tools and MCP servers across supported providers. Graph workflows add checkpointing and human-in-the-loop control. Microsoft/Azure enterprise stacks and teams migrating from AutoGen or Semantic Kernel.

Layer 4

Low-code builders are orchestration products, not toys

n8n

Operational workflow automation with triggers, nodes, credentials, execution history, and AI agent nodes.

Langflow

Visual builder for agents and flows that makes component wiring inspectable and reusable.

Flowise

Visual LLM app builder with Agentflow for multi-step and agentic flows.

They add value when domain owners need visibility and iteration speed, or when the real work is connecting business systems.

Langflow and Flowise

Visual builders make composition legible

Product Boundary Support agent use

Flowise Visual LLM app and Agentflow builder. Wire ticket intake, policy retrieval, decision branches, and response generation.

Low-code boundary

The question is not low-code versus code

Low-code is useful for

  • Business-system integration.
  • Trigger and notification plumbing.
  • Fast operational prototypes.
  • Flows that domain experts must inspect.

Code is useful for

  • Typed contracts and tests.
  • Fine-grained state semantics.
  • Custom authorization and deployment patterns.
  • Deep observability and CI/CD ownership.

Layer 5

Databases are the agent's product memory, not just vector search

Database role What lives there Why it matters
Application state Tickets, orders, customers, refunds, approvals. The agent must act on source-of-truth records.
Retrieval index Policy chunks, embeddings, metadata, document versions. Ground responses in current product and policy knowledge.
Runtime state Threads, checkpoints, workflow snapshots, pending interrupts. Resume safely and explain what happened.
Learning system Traces, feedback, eval datasets, golden answers, regressions. Improve behavior with evidence instead of anecdotes.

MongoDB

MongoDB fits when operational data and semantic search should sit together

Support agent use

  • Customer profile, ticket timeline, order events, entitlement flags.
  • Policy chunks with embedded vectors and rich metadata filters.
  • Trace/audit records that match application document shape.

Boundary

MongoDB is not the agent harness. It is the durable data plane the harness and runtime query, update, and audit against.

Document data Vector Search Metadata filters Operational records

Postgres and pgvector

Postgres fits when relational integrity and vectors should share a transaction boundary

Support agent use

  • Refund approvals, account entitlements, SLA rules, and policy versions in relational tables.
  • Policy embeddings through pgvector alongside ordinary SQL joins and filters.
  • Workflow state and audit events with transactional guarantees.

Boundary

pgvector adds vector similarity search to Postgres; it does not replace retrieval design, chunking, reranking, or access-control filtering.

SQL Transactions Vector similarity Policy joins

Retrieval pipeline

RAG is a data product pipeline, not a one-line tool

Ingest Load policy docs, ticket histories, release notes, and procedures.
Transform Chunk, normalize, classify, redact, and attach metadata.
Index Store text, embeddings, metadata, version, and permissions.
Retrieve Filter by tenant, product, region, entitlement, freshness.
Evaluate Measure recall, citation quality, answer quality, and cost.

Memory taxonomy

Memory means several different things

Memory type Example Storage pattern
Conversation history Messages in ticket T-9482. Thread/session store, often scoped by run or user.
Working state Risk score, proposed action, pending approval. Workflow checkpoint or runtime state table.
Long-term user memory Customer prefers SMS updates and has approved exception handling. Profile facts with write policy, provenance, correction, deletion.
Semantic memory Similar past tickets and effective resolutions. Vector search over curated prior interactions.

Durable state

Checkpoints are different from memories

Checkpoint

Execution state needed to resume a specific run: graph state, pending node, tool outputs, interrupt payload, and retry position.

Memory

Context intentionally made available to future runs: facts, preferences, summaries, examples, or retrieved knowledge.

Storing both in the same database can be reasonable. Treating them as the same concept is where agent systems become hard to debug.

Layer 6

Observability turns agent behavior into inspectable evidence

Trace

Every model call, tool call, retrieval step, and branch.

Metrics

Latency, tokens, cost, retries, approvals, failure modes.

Feedback

Human ratings, corrections, escalations, user outcomes.

Evals

Regression tests, golden cases, judges, and scenario simulation.

LangSmith

LangSmith is the LangChain ecosystem control plane

What it adds

Tracing, monitoring, evaluation, datasets, feedback, prompt engineering, deployment support, automations, and integrations across LLM application frameworks.

Support agent use

Compare traces for refund errors, build a dataset of risky tickets, run evals before prompt changes, and monitor production quality by ticket category.

Langfuse

Langfuse is framework-independent LLM observability and evals

What it captures

Trace trees with prompts, responses, token usage, latency, cost, tool calls, retrieval steps, sessions, and metadata.

Why it matters

You can instrument a mixed stack: Vercel AI SDK UI, Mastra or LangChain harness, n8n automation, and custom database tools.

Observability becomes the shared record when a single user interaction crosses multiple frameworks.

Developer loop

A production agent team needs a release process

Design

Define tools, memory writes, approval points, and data ownership before prompts.

Test

Run unit tests for tools, fixture-based traces, and eval datasets for behavior.

Operate

Monitor cost, failures, user corrections, drift, policy changes, and rollback paths.

Frameworks help, but the product discipline is the difference between a demo and a dependable system.

Interop

Protocols reduce glue, but do not erase ownership

Interop surface What it helps with What you still own
MCP Expose tools and data sources to agent clients in a common way. Auth, permissions, rate limits, tool semantics, data leakage.
A2A Agent-to-agent communication across languages or platforms. Trust, identity, audit, task boundaries, failure handling.
OpenAPI and webhooks Turn existing business APIs and event streams into tools. Contract quality, retries, idempotency, and business invariants.

Safety boundary

Human approval is a runtime feature, not a prompt

Bad boundary

"Only issue refunds when appropriate" in the system prompt, with a tool that directly creates refunds.

Good boundary

The agent proposes a refund request. The workflow pauses. A manager approves, edits, or rejects. The tool executes only after deterministic authorization.

Pick by workload shape

Workload Good fit Reason
Web chat with streaming UI and tool cards Vercel AI SDK plus custom tools Product surface and streaming UX are central.

Blueprint

A realistic support agent architecture

UI Ticket console streams draft, citations, tool status, and approval forms. Vercel AI SDK UI or custom frontend.
Agent harness Classify, retrieve, call tools, draft response, emit structured action plan. Mastra, LangChain, OpenAI Agents SDK, ADK, or Microsoft Agent Framework.
Workflow runtime Approval, branch, retry, schedule follow-up, write audit, resume after human input. LangGraph, Mastra workflows, ADK workflows, Microsoft workflows, n8n.
Data plane Orders, customers, policy docs, embeddings, checkpoints, traces, eval datasets. MongoDB, Postgres/pgvector, object storage, analytics warehouse.
Ops Trace every run, run evals, observe cost/quality, version prompts and tools. LangSmith, Langfuse, Mastra observability, Agno AgentOS, provider telemetry.

Tooling value

What tooling is really buying you

Faster assembly

Provider adapters, tool schemas, retrievers, streaming, and UI helpers reduce boilerplate.

Safer boundaries

Typed tools, guardrails, middleware, interrupts, and workflow state make behavior inspectable.

Operational learning

Traces, evals, feedback, and datasets let the system improve without guessing.

The valuable tools do not make the model smart. They make the system surrounding the model explicit, testable, and recoverable.

Recap

The stack fits together when each layer owns one job

Harness

Owns the model loop, tools, context, and immediate behavior.

Runtime

Owns long-running state, branches, checkpoints, interrupts, and recovery.

Builder/control plane

Owns composition, operations, governance, and human visibility.

Database

Owns product truth, retrieval indexes, memory, checkpoints, audit, traces, and eval evidence.
