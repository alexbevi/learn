Agent Memory Systems

A technical and practical rebuild of agent memory from first principles: memory types, write paths, read paths, databases, framework primitives, managed services, emerging layers, and production governance.

85 minutes / 46 slides / Last updated 2026-05-27

Learning goals

Memory should become an architecture choice

Taxonomy

Separate working, session, episodic, semantic, procedural, graph, and shared memory.

Mechanics

Trace memory through extraction, consolidation, retrieval, prompt assembly, correction, and deletion.

Tooling

Compare LangChain, LangGraph, LangMem, Mastra, LlamaIndex, OpenAI Agents SDK, Mem0, Zep, Letta, Cognee, Memori, and emerging layers.

Operations

Evaluate memory for precision, freshness, privacy, observability, governance, cost, and product fit.

Running example

Customer account intelligence agent

What it does

  • Triages support and success requests across months.
  • Tracks commitments, preferences, incidents, renewals, and risk signals.
  • Retrieves contracts, tickets, product docs, usage data, and policy.
  • Coordinates specialist agents for billing, support, and account planning.

Why it stresses memory

The agent must remember enough to be useful, forget enough to be safe, and verify enough against product truth to avoid acting on stale or invented context.

First principle

Memory is not chat history

Chat history Raw sequence of turns. Useful for continuity, but noisy, token-heavy, and usually scoped to one thread.
Agent memory Selected and structured retained context with scope, provenance, lifecycle, retrieval, and policy.
Product truth Authoritative business data. Memory can summarize it, but should not silently replace it.

The practical question is not whether the agent can remember. It is what the agent is allowed to remember, who can see it, and how it is corrected.

Architecture

Memory has layers with different owners

The database layer is not just vector search. It is where product truth, memory records, indexes, and auditability become enforceable.

Scope

Pick scope before picking storage

ThreadOne conversation or workflow run.
UserPreferences and user-specific facts.
AccountBusiness relationship, incidents, commitments.
OrganizationShared policy, contracts, operating model.
AgentLearned behavior and procedures for one agent.
TeamReusable playbooks and handoff patterns.
FleetCross-agent memory with governance and visibility.
TenantHard isolation boundary for safety and compliance.

Taxonomy

Memory types solve different jobs

Working memory

Active task state belongs close to the run

For the account agent

Current customer, open ticket, active escalation, proposed refund, risk threshold, and partially completed tool results.

Design rule

Working memory should be compact, task-local, and easy to discard. It is not the place for long-term customer truth.

Session memory

Session memory preserves a thread

Typical implementation

Message history, checkpointed graph state, recent tool calls, and generated artifacts stored under a stable thread id.

Boundary

It helps the agent resume a conversation. It does not automatically become trusted long-term memory.

OpenAI Agents SDK sessions and LangGraph checkpointed state are good examples of session continuity mechanisms.

Long-term memory

Long-term memory survives the thread

SemanticFacts and concepts: account prefers email escalation after outages.
EpisodicEvents: on May 11, support promised a migration report by Friday.
ProceduralRules: for regulated accounts, require approval before refund offers.

Cross-thread memory must be scoped, versioned, and correctable because the agent will use it when the original conversation is gone.

Data model

A memory unit needs metadata

{
  "id": "mem_9W4",
  "type": "episodic",
  "subject": { "kind": "account", "id": "acct_42" },
  "scope": ["tenant:acme", "account:acct_42"],
  "content": "Support committed to send migration report by Friday.",
  "source": { "threadId": "thr_81", "traceId": "run_7J" },
  "confidence": 0.82,
  "validFrom": "2026-05-21",
  "validTo": null,
  "policy": { "pii": false, "retentionDays": 180 }
}

The fields around the content are what make memory governable.

Write path

Memory creation is a pipeline

Write timing

Hot path and background writes trade latency for freshness

Write modeUse whenRisk
Hot pathThe next response needs the memory immediately.Adds latency and can over-trust unverified information.
Background consolidationThe memory improves future sessions but not the current turn.May lag behind recent messages or miss a narrow correction.
Human-approvedThe memory affects policy, money, compliance, or sensitive preferences.Requires workflow and UX support.

Admission control

The hardest write decision is what not to remember

Reject noiseTransient greetings, jokes, and one-off wording.
Reject weak factsGuesses, hallucinated details, and unverified inferences.
Reject risky dataSensitive personal data without consent or policy basis.
Reject duplicatesRepeated facts that should update an existing record.

A memory system that remembers everything becomes a slower, more confident transcript search engine.

Consolidation

Consolidation turns memories into a model

MergeCombine duplicate preferences and related observations.
SupersedeMark old facts inactive when new evidence replaces them.
SummarizeCompress long histories into dense, source-linked observations.
LinkConnect people, accounts, tickets, incidents, and procedures.
ExpireRemove time-bound operational details after their value drops.
EscalateRequire review when memory affects risk, spend, or compliance.

Indexing

One memory needs several retrieval surfaces

SurfaceQuestion it answersExample
Exact lookupWhat do we know about this account?Fetch by accountId.
Semantic indexWhat prior issue is similar?Vector search across incidents.
GraphWho is connected to this decision?Traverse account, owner, ticket, policy.
RecencyWhat changed recently?Rank by freshness and validity.
AuditWhy did the agent use this?Trace memory id to source run.

Read path

Retrieval should be planned, not accidental

Ranking

Vector score is only one signal

RelevanceDoes the memory answer this decision?
FreshnessIs it still valid for the account?
ConfidenceWas the memory extracted from strong evidence?
ScopeIs this agent allowed to see it?
SourceDid it come from user, tool, policy, or inference?
RiskCould misuse cause harm or compliance issues?
CostIs it worth prompt tokens now?
ContradictionDoes a newer record supersede it?

Prompt budget

The memory system earns value by omitting context

Bad memory assembly

Injects broad summaries, stale facts, every similar episode, and unrelated user preferences because they look semantically close.

Good memory assembly

Injects the minimum set of scoped, fresh, source-linked memory records needed for the next decision.

Long context does not remove the need for memory selection. It just moves the failure from missing context to distractor context.

Freshness

Memory must support contradiction and change

SupersededThe customer changed their preferred escalation channel.
ExpiredA migration freeze was relevant last month, but not now.
ContestedA user says the agent remembered the wrong commitment.

Treat remembered context as guidance. For irreversible actions, verify against source systems or ask for confirmation.

Databases

Databases provide distinct memory roles

MongoDB framing

MongoDB's useful framing is memory engineering

Conceptual model

Memory types, memory units, application modes, and persistent memory management around the LLM.

Architecture value

Flexible documents, metadata, vector search, operational data, and audit records can live near each other without forcing memory into one shape.

The useful lesson is broader than one product: memory needs a record model and retrieval model that can evolve together.

Postgres and pgvector

Relational memory fits identity-heavy systems

Strong fitUser identity, account membership, retention policy, approvals, and transactional workflows.
Vector extensionUse pgvector when semantic recall belongs close to relational records.
WatchoutDo not force graph-like or flexible document memory into brittle tables too early.

Graph memory

Graph memory helps when relationships matter

Good fit

Temporal claims, account relationships, support incidents, people, ownership, commitments, dependencies, and multi-hop questions.

Operational risk

Graph construction can produce noisy edges. The system needs provenance, edge types, confidence, and update rules.

LangChain ecosystem

Separate checkpoints from long-term stores

LangChain agent stateConversation history in agent state, persisted through a checkpointer.
LangGraph storesCross-thread memory in custom namespaces for user or application data.
Design implicationResume state and durable memory are different persistence contracts.

Deep Agents and LangMem

Memory is moving toward explicit tools and files

Deep Agents

Persistent memory can be represented as scoped files, with agents reading and updating memory through file operations and consolidation.

LangMem

Memory tools can be added to LangGraph applications; production setups need persistent stores rather than in-process memory.

Mastra

Mastra treats memory as a first-class agent option

Message historyRecent conversation continuity.
Observational memoryBackground agents compress old messages into observations.
Working memoryStructured user data such as names, preferences, and goals.
Semantic recallRetrieve relevant past messages by meaning.

Its practical control points are resource, thread, memory processors, storage providers, and tracing.

LlamaIndex

LlamaIndex exposes memory as a component

memory = Memory.from_defaults(
    session_id="account-acme",
    token_limit=40000,
)

agent = FunctionAgent(llm=llm, tools=tools)
response = await agent.run(
    "Summarize the current renewal risk.",
    memory=memory,
)

The important abstraction is explicit memory injection and retrieval around agent execution.

OpenAI Agents SDK

Sessions provide conversation continuity

agent = Agent(
    name="Account assistant",
    instructions="Use session context, then verify facts with tools.",
)

session = SQLiteSession("account_acme_thread_42")

result = await Runner.run(
    agent,
    "What did we promise after the outage?",
    session=session,
)

Session memory stores conversation history. Durable customer memory still needs a deliberate data contract.

Dedicated services

Managed memory shifts work out of your app

ExtractionDecide what should become memory.
RetrievalFind the right memories for the next run.
GraphingLink entities, events, and time.
OperationsInspect, edit, delete, evaluate, and monitor memory.
IntegrationSDKs, APIs, MCP, and framework adapters.
TradeoffCost, lock-in, data boundary, and explainability.

Mem0

Mem0 positions memory as managed infrastructure

What the docs emphasize

User, agent, and session memory behavior; add, search, update, and delete operations; graph memory, async clients, rerankers, metadata filters, and MCP integration.

How to evaluate it

Test memory admission quality, retrieval precision, deletion semantics, tenant isolation, latency, and cost on real account-agent tasks.

Comparison

Memory tooling differs by ownership model

Zep

Zep emphasizes temporal graph memory

High-level API

Add memory, retrieve memory, and pass the returned context into the next LLM call.

Important nuance

Zep recommends including the last few raw messages as short-term memory because ingestion and graph-derived context may lag recent turns.

Letta

Letta makes agent state explicit and editable

Persisted stateMessages, reasoning, tool calls, memory, and state are persisted in a database.
Memory blocksImportant core memories are injected into context and can be edited by tools or API.
Shared blocksBlocks can attach to multiple agents when memory should be shared intentionally.

Cognee and Memori

Memory systems are expanding beyond transcript recall

Cognee

Exposes remember, session memory, graph memory, and an agent-memory decorator that retrieves memory at an async function boundary.

Memori Labs

Frames memory as enterprise memory fabric with multi-agent processing, dual memory modes, and provider configuration.

Emerging layers

Rippletide, Semvec, and Caura are signals, not settled standards

LayerSignalConfidence
RippletideDecision runtime concepts: reinforcement infrastructure between agents and real-world action.Medium
SemvecFixed-size semantic state vector plus tiered content-aware memory for constant-cost context.Low
CauraGoverned shared memory for fleets where visibility, policy, and security boundaries matter.Low

Privacy

Memory creates retained data

ConsentIs the agent allowed to store this?
ScopeWho can retrieve it later?
SensitivityDoes it contain personal or regulated data?
DeletionCan the user or admin remove it?
AuditCan you prove how it was used?
StalenessCan a correction supersede it?
IsolationCan tenants leak through shared memory?

Governance

Governed memory has a lifecycle

Evals

Evaluate writes and reads separately

Eval targetQuestionsDataset examples
Write qualityDid it remember the right thing, with the right type and scope?Conversations with known preferences, incidents, corrections, and non-memory noise.
Read usefulnessDid it retrieve the minimum useful context for the task?Account-risk tasks requiring prior commitments and policy verification.
SafetyDid it avoid forbidden, stale, or cross-tenant memory?Tenant boundary tests, deletion tests, prompt-injected memory poisoning.

Observability

Memory traces should explain both selection and omission

Write traceCandidates, rejected records, policy gates, extracted fields, and consolidation decisions.
Read traceQuery plan, candidate ids, scores, policy filters, final prompt context, and token budget.
Outcome traceWhether the selected memory improved task success or caused a correction.

Failure modes

Memory failures are predictable

Over-rememberingNoise crowds out signal.
Stale factsOld memory beats current truth.
False positivesSemantic similarity retrieves the wrong episode.
Memory poisoningUntrusted input becomes trusted memory.
Tenant leakageShared memory crosses isolation boundaries.
Hidden lock-inNo export, audit, or correction path.
Runaway costContext keeps growing without measurable value.
Unclear authorityThe agent treats memory as source of truth.

Recipe

A practical build sequence

1. Product truthDefine authoritative records, ids, permissions, and tools.
2. SessionsAdd thread continuity and resumable workflow state.
3. Scoped memoriesAdd typed user/account memory with correction and deletion.
4. RetrievalAdd text, vector, graph, and recency indexes only where needed.
5. EvalsMeasure write, read, safety, and outcome quality.
6. ServicesAdopt dedicated or governed memory when failures justify the operational tradeoff.

Decision framework

Choose the smallest strategy that satisfies correctness

Code-level shape

Own the port even when a framework owns the implementation

type MemoryQuery = {
  subject: { kind: "user" | "account" | "agent"; id: string };
  scope: string[];
  task: string;
  allowedTypes: MemoryType[];
  traceId: string;
};

interface AgentMemory {
  remember(candidate: MemoryCandidate): Promise<MemoryWriteResult>;
  recall(query: MemoryQuery): Promise<RankedMemory[]>;
  correct(id: string, patch: MemoryCorrection): Promise<void>;
  delete(scope: string[], reason: string): Promise<void>;
}

The interface forces scope, provenance, policy, and traceability to remain visible.

Recap

Good memory is typed, scoped, governed, and evaluated

Architecture stance

Memory is a product data system around the model. It interacts with runtime state, product truth, retrieval indexes, and governance.

Practical stance

Start with sessions and product truth, add long-term memory deliberately, and adopt specialized layers only when they solve measured failures.
