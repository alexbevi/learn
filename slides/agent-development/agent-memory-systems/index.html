Agent Memory Systems

A technical and practical tour of memory abstractions, storage patterns, framework implementations, memory services, graph layers, and governance models for long-running agents.

85 minutes / 50 slides

Learning goals

Agent memory should stop feeling like a vague feature

Taxonomy

Separate working, session, episodic, semantic, procedural, associative, profile, and shared memory.

Implementation

Design write paths, read paths, indexes, scopes, TTLs, contradiction handling, and audit trails.

Tooling

Compare framework-native memory with Mem0, Letta, Zep, Memori Labs, Cognee, Rippletide, Semvec, and Caura.

Architecture

Know where MongoDB, Postgres, vector stores, graph stores, and managed memory services fit.

Running example

Long-running customer account agent

What the agent does

  • Handles support tickets across months.
  • Remembers preferences, unresolved commitments, and prior incidents.
  • Retrieves policies, contracts, order history, and product facts.
  • Escalates risky refunds and tracks what actions already happened.

Why memory matters

Without memory, the agent is a stateless responder. With badly designed memory, it becomes confidently stale. With governed memory, it becomes an accountable product system.

Core thesis

Memory is a data system around the model

Encode Turn raw interaction into structured, searchable, permissioned memory.
Store Persist memory units, embeddings, graph edges, provenance, and lifecycle state.
Retrieve Select what matters for this task, user, session, and risk boundary.
Synthesize Inject memory as compact, sourced context rather than raw database dumps.
Govern Correct, expire, delete, audit, and evaluate memories over time.

MongoDB framing

Agent memory is a computational exocortex

Useful framing

MongoDB frames agent memory as a memory-management system that integrates context-window state with persistent encoding, storage, retrieval, and synthesis of experiences.

Why it helps this deck

It keeps memory broader than vector search: memory units need temporal behavior, metadata, relationships, and actionability.

Working memory

Working memory belongs to the active run

What it stores

  • Current goal and subgoals.
  • Tool observations and intermediate conclusions.
  • Risk flags, branch decisions, pending approvals.

Failure mode

If working state is hidden in conversation text, retries and approvals become fragile. Store it as graph state or workflow state when correctness matters.

Session memory

Session memory is continuity, not learning

Message window

Recent chat turns that fit in the context budget.

Summaries

Compressed prior turns when the thread is too long.

Session store

OpenAI Agents SDK sessions, LangGraph thread checkpoints, and app-owned chat history.

Session memory answers: "What have we been discussing?" Long-term memory answers: "What should future runs know?"

Episodic memory

Episodic memory is evidence of what happened

Ticket Customer asked for a refund.
Investigation Shipment delayed by carrier.
Decision Manager approved credit.
Action Credit issued and email sent.
Outcome Customer retained account.
Lesson Escalate carrier delays above $500.

Semantic memory

Semantic memory stores facts, but facts need ownership

Good semantic memories

  • Small enough to verify.
  • Scoped to a subject.
  • Linked to source events.
  • Timestamped and correctable.

Bad semantic memories

  • Model guesses saved as facts.
  • Summaries without provenance.
  • Preference facts shared across tenants.
  • Contradictions with no lifecycle.

Procedural memory

Procedural memory is how the agent learns to work

Pattern Example Storage
Playbook High-value refund process Read-only docs or policy database
Skill How to write an escalation note Versioned skill file or prompt module
Learned strategy Ask for carrier tracking before refund Curated memory or eval-approved rule

Shared memory

Multi-agent systems need scoped sharing

Private

Agent scratch state, draft reasoning, unresolved hypotheses.

Team

Support playbooks, product gotchas, recent incident lessons.

Organization

Policies, compliance facts, approved procedures, tenant-safe knowledge.

Caura and MemClaw explicitly position around governed shared memory for agent fleets.

Memory unit

Design the record before designing retrieval

type MemoryUnit = {
  id: string;
  subject: { type: "customer" | "account" | "agent" | "policy"; id: string };
  scope: { tenantId: string; visibility: "private" | "team" | "org" };
  kind: "preference" | "event" | "fact" | "procedure" | "relationship";
  content: string;
  source: { type: "ticket" | "tool" | "admin" | "document"; id: string };
  confidence: number;
  validFrom: string;
  validUntil?: string;
  supersedes?: string[];
  embedding?: number[];
  graphEdges?: Array<{ predicate: string; objectId: string }>;
};

Write path

The write path is where most memory quality is won

Capture Message, tool result, workflow event, document, correction.
Filter Should this be remembered? Is it allowed?
Extract Structured fact, event, preference, relationship, procedure.
Reconcile Merge, supersede, invalidate, dedupe, link to source.
Index Embedding, graph edge, keyword, metadata, temporal index.
Audit Who wrote it, why, from what evidence, with what policy?

Read path

Recall is routing, retrieval, and context assembly

Need What does this task require?
Scope User, account, ticket, org, agent, time.
Query Vector, graph, SQL, keyword, profile lookup.
Rank Relevance, recency, confidence, authority, risk.
Compose Compact context with sources and caveats.
Inject System prompt, tool result, state, or UI review.

Lifecycle

Memory needs contradiction and forgetting semantics

Active

Available for recall and prompt injection.

Superseded

Kept for audit, not used as current truth.

Expired

Hidden after TTL or policy validity ends.

Deleted

Removed by privacy request or retention policy.

Governance

Memory is a privacy and authorization surface

Risk Control Implementation hook
Silent PII storage Classify and redact before write Extraction policy, DLP, human approval
Cross-tenant leakage Namespace and row-level isolation Tenant-scoped query filters
Stale preferences Validity windows and source rank Temporal metadata
Wrong memories Correction and supersession Admin UI, audit log, feedback loop

Evaluation

Memory quality needs its own test suite

Recall

Did the right memories appear?

Precision

Did irrelevant memories stay out?

Update

Did contradictions supersede old facts?

Outcome

Did memory improve task success without new risk?

Treat memory as a behavior-changing subsystem: trace reads and writes, label failures, and run regression scenarios.

MongoDB implementation

Document memory fits MongoDB's shape well

Natural fit

  • Memory units with flexible metadata.
  • Account/ticket/customer documents.
  • Vector Search over memory content and policy chunks.
  • Aggregation and change streams for consolidation workflows.

Design discipline

Keep application truth, memory units, retrieval indexes, and agent traces logically distinct even when they share one database platform.

Postgres implementation

Postgres and pgvector fit transactional memory

Good fit

  • Relational account, policy, approval, and audit data.
  • pgvector for embeddings next to SQL filters.
  • Transactional writes for action memory and workflow state.

Watch out

Vector similarity alone does not solve memory lifecycle, contradiction handling, provenance, or governed sharing.

TypeScript sketch

A minimal app-owned memory write path

async function rememberPreference(input: {
  tenantId: string;
  customerId: string;
  sourceTicketId: string;
  preference: string;
}) {
  await policy.assertCanStoreMemory(input.tenantId, "customer_preference");

  const memory = {
    subject: { type: "customer", id: input.customerId },
    scope: { tenantId: input.tenantId, visibility: "team" },
    kind: "preference",
    content: input.preference,
    source: { type: "ticket", id: input.sourceTicketId },
    confidence: 0.9,
    validFrom: new Date().toISOString(),
    status: "active",
  };

  await memories.insertOne(memory);
  await memoryIndex.enqueue({ memoryId: memory.id });
}

Python sketch

A minimal recall path separates query from injection

async def recall_for_ticket(customer_id: str, ticket_text: str) -> str:
    candidates = await memory_store.search(
        subject={"type": "customer", "id": customer_id},
        query=ticket_text,
        filters={"status": "active", "visibility": {"$in": ["team", "org"]}},
        limit=12,
    )

    ranked = rank_by_policy(candidates, weights={
        "semantic": 0.45,
        "recency": 0.2,
        "confidence": 0.2,
        "source_authority": 0.15,
    })

    return compose_context(ranked[:5], include_sources=True)

LangChain and LangGraph

LangChain separates thread memory from long-term stores

Short-term

LangGraph checkpoints persist thread-scoped state: messages, graph state, pending interrupts, and resumable execution.

Long-term

LangChain long-term memory is built on LangGraph stores: JSON documents organized by namespace and key, searchable across conversations.

Strong pattern: use checkpoints for run durability, stores for cross-thread memory, and tools for explicit memory writes.

LangMem

LangMem makes memory extraction and management explicit

Semantic

Facts, profile fields, and searchable knowledge collections.

Episodic

Past examples, summaries, outcomes, and experience traces.

Procedural

Rules, style, skills, and learned behavior patterns.

LangMem's useful design question: what should be learned, when should it be formed, and where should it be stored?

Mastra

Mastra treats memory as part of the TypeScript agent framework

Thread history

Conversation continuity scoped by memory.thread and memory.resource.

Semantic recall

Vector-backed recall over prior memories, with explicit embedder configuration.

Observational memory

Newer compression-oriented memory mode for long-running context.

Mastra is compelling when agent code, memory, RAG, workflows, and product app code are all TypeScript.

LlamaIndex

LlamaIndex memory reflects its data-first DNA

Memory class

Combines short-term FIFO chat history with optional long-term memory blocks that receive flushed messages and process them.

Best fit

Agents whose hardest memory problem is retrieval over documents, histories, and structured knowledge rather than workflow checkpointing.

CrewAI and OpenAI Agents SDK

Other frameworks expose narrower but useful memory surfaces

Framework Memory shape Boundary
CrewAI Unified Memory API with remember, recall, scope, recency, importance, and background saves. Good for multi-agent crews and flows where shared work memory matters.
OpenAI Agents SDK Sessions store conversation history across agent runs. Good session continuity; long-term domain memory still often belongs in an app store or memory service.

Dedicated services

Managed memory services move memory out of the harness

Mem0 Universal memory layer with managed and open-source options.
Zep High-level Memory API plus graph API and user-level knowledge graph.
Letta Memory-first stateful agents with persistent identity and memory blocks.
Memori Labs Agent-native memory infra with structured extraction and trace memory.
Cognee Graph/RAG memory layer for agent memory and knowledge organization.

Mem0

Mem0 optimizes for easy persistent memory across apps

What it abstracts

  • Adding memories from conversations.
  • Searching memories by user, agent, or session.
  • Managed platform or self-hosted open source deployment.
  • Integrations with LangChain, CrewAI, Vercel AI SDK, and others.

Best fit

Fast path to cross-session user memory when you do not want to design extraction, indexing, and recall from scratch.

Zep

Zep is graph-backed context engineering for agents

Memory API

Add chat history with one call, retrieve a context string with one call, and use recent messages as short-term memory while the context string represents long-term memory.

Graph API

Use lower-level graph operations when you need business data, custom memory construction, group graphs, or retrieval customization.

Letta

Letta starts from stateful agents, not stateless threads

Core idea

Letta agents maintain persistent memory and evolve over interactions. The agent is the persistent entity, not a disposable chat thread.

Design lesson

Memory can be a first-class agent architecture: identity, memory blocks, tools, and context management shape the agent's behavior over time.

Memori Labs

Memori Labs frames memory as agent-native infrastructure

Attribution

Memory is scoped to entities, processes, sessions, and agent execution context.

Extraction

Interactions are captured and asynchronously distilled into structured memories.

Recall

Supports automatic and manual recall, plus trace/execution memory for agents.

Treat as an emerging service: validate APIs, deployment model, retention controls, and integration depth before committing.

Cognee

Cognee is memory as graph-organized knowledge

What it adds

A memory layer for agents that builds structured knowledge from data, then exposes retrieval over graph and semantic context.

Best fit

Data-heavy agents that need knowledge graph organization, provenance, and repeatable memory construction from documents and interactions.

Rippletide

Rippletide positions memory as a context graph and decision layer

Context Graph

Persistent graph memory for facts, decisions, preferences, context, relationships, and invalidation when information changes.

Enterprise angle

Rippletide describes itself as a decision runtime/database between an agent and the real world, with MCP-oriented access and hallucination/evaluation positioning.

Semvec

Semvec represents the compact semantic-state direction

Emerging idea

Semvec-style layers emphasize vector-addressed semantic state: compact memory representations that can be searched and updated without replaying entire chat histories.

Practical caution

Treat Semvec as an emerging/experimental option. Evaluate maintenance, API stability, persistence model, and how contradictions are resolved.

Caura and MemClaw

Caura focuses on governed shared memory for agent fleets

Caura framing

A shared memory layer for agents to remember, coordinate, and comply across LLMs, teams, tools, and enterprise interfaces.

MemClaw framing

Governed memory with visibility scopes, trust tiers, audit, tenant isolation, PII quarantine, lifecycle automation, and MCP-style tools.

Comparison

Compare by boundary, not by feature list

Layer Primary boundary Strong when Watch for
Framework memory Inside agent/runtime You want simple integration and fewer services Portability, observability, hidden write policy
App-owned DB Your product data plane Memory must align with app truth and auth Building extraction/ranking/governance yourself
Managed memory service External memory API You want fast persistent memory and integrations Vendor lock-in, privacy model, opaque ranking
Graph/decision layer Relationships and validity Multi-hop, temporal, governed decisions matter Operational complexity and graph hygiene

Architecture choices

Four patterns cover most practical systems

Embedded

Use the framework's memory primitives.

Database-backed

Own memory units, indexes, and lifecycle.

Service-backed

Call Mem0, Zep, Letta, Memori, or similar.

Graph-governed

Use graph/fleet layers for relationships and compliance.

Blueprint

Support agent memory architecture

Runtime state Ticket workflow state, tool outputs, pending approval, retries, current task scratchpad. LangGraph checkpoints, Mastra workflows, app workflow DB.
User/account memory Preferences, prior incidents, commitments, customer-specific constraints. MongoDB/Postgres memory units, Mem0, Zep, Letta, Memori.
Knowledge memory Policies, product docs, contracts, procedures, examples. RAG index, LlamaIndex, Cognee, vector/graph store.
Decision memory What action was taken, who approved it, why, and what changed later. Audit log, graph layer, Rippletide-style decision context.

Build path

How I would implement the first production version

1. Split state Runtime state, memory units, app truth, and traces.
2. Define schema Subject, scope, source, confidence, lifecycle, permissions.
3. Add tools Explicit remember, recall, forget.
4. Add extractor Background memory extraction from conversations and outcomes.
5. Add evals Recall, precision, contradiction, privacy, task outcome.
6. Add UI Let humans inspect, correct, delete, and approve memories.

Anti-patterns

What usually goes wrong

Everything is memory

Chat logs, policy docs, tool outputs, and user preferences all land in one vector index with no lifecycle.

The model silently writes truth

Unverified guesses become durable facts and later appear as authoritative context.

Recall is invisible

No trace shows which memories influenced the answer, so debugging depends on guesswork.

No correction path

Users can correct the answer but not the memory that caused the answer.

Write strategy

Save everything is not the same as remember wisely

Strategy Pros Cons
Raw transcript Complete evidence, easy audit Noisy, expensive, weak recall
LLM extraction Structured, compact, useful Can hallucinate, needs validation
Tool-mediated write Agent must choose to remember Misses facts unless prompted/evaluated
Human-approved memory High trust and governance Slower, higher UI burden

Database fit

Databases add value because memory is product data

MongoDB

Flexible memory units, operational records, metadata filters, Vector Search, and event-driven consolidation.

Postgres

Transactional memory, relational policy joins, audit, approval flows, and pgvector.

Graph stores

Entities, relationships, temporal validity, provenance, and multi-hop questions.

Vector search is one index. The memory system is the contract around what is allowed to become durable context.

Interop

Memory is becoming a protocol surface

MCP memory tools

Tools like remember, recall, and relate let assistants access memory services without framework-specific plugins.

Portability pressure

As agents move across IDEs, chat apps, runtimes, and vendors, memory ownership and transfer become product and governance questions.

Testing

Memory scenarios should look like product tests

scenario: "customer preference supersession"
turns:
  - user: "Send all updates by SMS."
    expect_memory_write:
      kind: preference
      content: "prefers SMS updates"
  - user: "Actually, email is better for billing issues."
    expect_memory_update:
      active: "prefers email for billing issues"
      retained: "prefers SMS updates for other updates"
assertions:
  - recall("billing contact preference") includes "email"
  - recall("shipping update preference") includes "SMS"
  - answer cites memory source ticket ids

Decision matrix

Pick memory tooling by the dominant risk

Dominant need Start with Why
Quick personalization Mem0 or Zep Memory API Fast persistent recall with framework integrations.
Memory-first agents Letta Agent identity and persistent memory are central abstractions.
TypeScript app agents Mastra memory plus app DB Agent memory stays close to product code and workflows.
LangChain ecosystem LangGraph stores plus LangMem Strong split between checkpoints, stores, and memory extraction.

Recap

The useful mental model

Memory is not RAG

RAG retrieves external knowledge. Memory changes what the agent knows about users, work, itself, and prior outcomes.

Memory is not a transcript

Transcripts are evidence. Memory is structured, scoped, lifecycle-managed context derived from evidence.

Memory is not only storage

The hard parts are write policy, contradiction handling, retrieval timing, governance, and evaluation.

Memory is product infrastructure

It needs schemas, permissions, audit, correction, deletion, observability, and tests.
