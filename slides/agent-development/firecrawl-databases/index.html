Firecrawl, Databases, and MongoDB

How web acquisition becomes AI-ready data, where databases fit, and when MongoDB is a strong architectural choice.

55 minutes / 36 slides

Why this matters

AI systems need web context, but the web is not a database

Pages are messy

Navigation, ads, cookie banners, scripts, redirects, PDFs, tables, and hidden state all pollute naive scraping.

Sources change

Docs, prices, policies, competitors, and changelogs drift after the first successful ingest.

Models need evidence

Useful answers require source lineage, freshness, retrieval quality, and repeatable ingestion behavior.

Firecrawl handles acquisition. Your data architecture decides what becomes durable, searchable, and trustworthy.

Learning goals

By the end, you should be able to explain

Practical use

Choose between scraping, mapping, crawling, searching, browser interaction, and structured extraction.

Technical architecture

Design a web-to-RAG pipeline with job state, raw capture, normalized documents, chunks, embeddings, and retrieval.

Database judgment

Separate source inventory, job ledger, archive, document store, vector index, evals, and audit logs.

Strategic fit

Decide when Firecrawl is worth adopting and when MongoDB is the right place to keep the acquired data.

Endpoint map

Pick the endpoint based on what you know

scrapeOne known URL. Return markdown, JSON, HTML, links, screenshot, or metadata.
batch scrapeMany known URLs. Same extraction shape, async delivery, webhooks, and concurrency controls.
mapDiscover URLs first. Use sitemap, filters, subdomains, and query handling before fetching content.
crawlWalk a site from a base URL with include/exclude paths, depth, limits, webhooks, and nested scrape options.
searchStart with a query. Optionally scrape search results and use domain, time, and category filters.
interactResume a browser session after scrape and click, type, scroll, or run Playwright-style code.

Output contract

Firecrawl gives you acquisition artifacts, not final truth

ContentMarkdown, HTML, raw HTML, screenshots, parsed documents, or structured JSON.
MetadataSource URL, final URL, title, description, language, status code, and page-level fields.
Job factsJob id, status, total pages, completed pages, credits used, expiration, and pagination cursor.
Failure factsValidation errors, timeouts, rate limits, concurrency queue data, page errors, and retry signals.

Your database decides what is canonical, current, queryable, and auditable.

Scrape

scrape is the single-page primitive

Use it when

  • You know the exact URL.
  • You need synchronous output.
  • You want controlled markdown or JSON extraction.
  • You need cache age, waits, headers, mobile mode, or parser choices.

Be careful when

  • The URL list is unknown.
  • The page requires multi-step navigation.
  • The source changes frequently.
  • The page is only one piece of a larger corpus.

For one known page, start here before reaching for crawling or autonomous research.

Scrape code

Small call, meaningful acquisition contract

import { Firecrawl } from "firecrawl";

const firecrawl = new Firecrawl({ apiKey: process.env.FIRECRAWL_API_KEY });

const page = await firecrawl.scrape("https://docs.example.com/pricing", {
  formats: ["markdown"],
  onlyMainContent: true,
  maxAge: 60 * 60 * 1000,
  timeout: 120000,
});

await documents.updateOne(
  { sourceUrl: page.metadata?.sourceURL },
  {
    $set: {
      markdown: page.markdown,
      finalUrl: page.metadata?.url,
      title: page.metadata?.title,
      acquiredAt: new Date(),
    },
  },
  { upsert: true },
);

The important design move is making freshness, content shape, and provenance explicit.

Discovery

Map before you crawl when scope is uncertain

Map siteCollect candidate URLs from sitemap and discovered links.
→
FilterKeep docs, pricing, changelog, or product paths; drop login, tags, and low-value pages.
→
DedupeNormalize query parameters and keep stable source identifiers.
→
AcquireBatch scrape selected URLs or crawl with explicit path rules.

Discovery is an architectural control point for quality and cost.

Crawl lifecycle

Crawls are async jobs with page-level outcomes

SubmitBase URL, limits, sitemap mode, include/exclude paths, scrape options, webhook.
Queued or runningConcurrency and queue wait can affect timeout behavior.
Page eventEach crawled page can arrive through webhook or status polling.
StatusTrack total, completed, credits used, status, expiration, and next page cursor.
CompletePersist page data, metadata, source lineage, and acquisition settings.
Fail or retryClassify retryable errors separately from policy or validation errors.

Search

Use search when the source set starts as a question

Starting point Firecrawl shape Storage implication
Known URL scrape or batch scrape Store source policy and refresh cadence.
Known domain map then crawl Store URL inventory and inclusion rules.
Unknown source search or agent Store discovered URLs before relying on the answer.

Dynamic pages

interact is for pages that need browser state

ScrapeCall /scrape and keep the returned scrapeId.
→
InteractPrompt or run code to click, type, scroll, inspect, or navigate.
→
ExtractRead output, screenshot, live view, stdout, result, or error fields.
→
StopEnd the browser session when acquisition is done.

Browser sessions are powerful, but they are slower and more operationally sensitive than static acquisition.

Structured extraction

Choose extraction by URL uncertainty

Need Likely choice Why Watch out
One known page scrape JSON mode Controlled, synchronous, predictable. Still validate extracted fields.
Many known pages batch scrape or constrained agent Good for lists of product pages, docs, or known companies. Needs job ledger and dedupe.
Unknown sources agent or search Discovery is part of the task. Costs and repeatability need guardrails.

Running example

Build a competitor-docs assistant

Question

What changed in a competitor's API, pricing, and integration docs this week, and how should our product team react?

System responsibilities

  • Discover and classify documentation URLs.
  • Acquire current content and track changes.
  • Store source lineage, chunks, and embeddings.
  • Answer with citations and freshness.

Architecture

The pipeline is bigger than the scrape call

NormalizeClean content, extract fields, hash versions, split documents.
DatabaseJobs, sources, raw archive, documents, chunks, vectors, evals.

Database roles

Name the storage job before choosing the product

Source inventoryWhich sites, paths, owners, policies, and refresh rules exist?
Job ledgerWhat ran, failed, retried, cost credits, or needs replay?
Raw archiveWhat evidence was captured at a point in time?
DocumentsWhat cleaned content and extracted fields power the app?
Chunks and vectorsWhat text units are embedded and searchable?
Evals and auditWhat did the system retrieve, answer, cite, approve, and change?

Job ledger

Async acquisition needs local truth

RequestedSource id, endpoint, options, schema version, requested URL.
SubmittedFirecrawl job id, created time, expected delivery path.
Page receivedSource URL, final URL, status code, page hash, metadata.
NormalizedCleaned document, extraction output, lineage, policy checks.
IndexedChunks, embeddings, vector index, text index, filter metadata.
Retry or dead letterRetryable status, backoff time, permanent reason, operator note.

Webhooks and polling should converge on the same idempotent records.

Raw archive

Keep the evidence that produced the answer

Immutable versions

Store content hash, acquired time, Firecrawl options, source URL, final URL, and parser settings.

Replayability

When extraction rules improve, reprocess old evidence without pretending it was newly acquired.

Auditability

When an answer is challenged, trace it back to the exact page version and retrieval context.

Retrieval prep

RAG adds chunk and embedding state

DocumentNormalized markdown plus source lineage.
→
ChunksStable positions, headings, token counts, parent document id.
→
EmbeddingsVector field, model name, model version, dimensions.
→
RetrieverVector similarity, text search, filters, reranking, citations.

A chunk without provenance is just text with amnesia.

MongoDB fit

MongoDB is strong when content and metadata belong together

Role MongoDB fit Why
Normalized documents Strong Scraped pages, metadata, extracted fields, lineage, and policy data are naturally document-shaped.
Chunks with vectors Strong Vector fields, metadata filters, and application fields can live in one collection model.
Job ledger Strong Flexible records work well for endpoint-specific options, page events, retries, and per-source state.
Raw binary archive Possible Often better in object storage, with MongoDB storing pointers and metadata.
Graph reasoning Depends Entity graphs may justify a graph database when relationship traversal is the core workload.

Document shape

MongoDB can keep source lineage beside retrieval data

{
  "_id": "chunk_doc123_0007",
  "docId": "doc_competitor_pricing_2026_05_31",
  "sourceUrl": "https://www.example.com/pricing",
  "headingPath": ["Pricing", "Enterprise"],
  "text": "Enterprise customers can request custom limits...",
  "embedding": [0.012, -0.034, 0.091],
  "metadata": {
    "company": "ExampleCo",
    "pageType": "pricing",
    "acquiredAt": "2026-05-31T14:05:00Z",
    "contentHash": "sha256:..."
  }
}

The useful MongoDB pattern is not only vector storage. It is vector storage plus product metadata and lineage.

MongoDB retrieval

Use semantic, full-text, and hybrid retrieval deliberately

Chunk documentsText, embedding, source metadata, policy fields, timestamps.
Vector searchSemantic similarity over embedding fields with optional pre-filters.
Full-text searchExact terms, product names, API names, errors, and phrase matching.
Hybrid fusionCombine vector and text results when both meaning and exact language matter.
Answer layerRetrieve, cite, summarize, compare, and log feedback.

Freshness

Monitoring turns crawling into a delta system

SameNo downstream reprocessing unless freshness metadata needs update.
NewCreate document, chunk, embed, index, and alert if relevant.
ChangedVersion content, diff, re-extract fields, refresh affected chunks.
RemovedTombstone or de-rank stale chunks instead of silently deleting evidence.
ErrorRetry, escalate, or mark source health degraded.

Freshness is a data lifecycle, not a cron job pasted onto ingestion.

Quality loop

Successful API calls are not the same as useful answers

CoverageDid we acquire the pages that matter?
CleanlinessDid boilerplate, nav, and duplicate content stay out?
ExtractionDo structured fields match human-reviewed truth?
RetrievalDo relevant chunks appear in top results with good filters?
CitationsCan each answer point to current source evidence?
CostWhat is cost per useful document and per useful answer?

Governance

Web acquisition has policy surfaces

Source policy

Allowed paths, blocked paths, owners, robots posture, legal notes, and retention class.

Credential boundary

Headers, cookies, authenticated sessions, and browser profiles should be scoped and auditable.

Data retention

Decide what raw content, extracted data, screenshots, and traces are retained or discarded.

Reliability

Rate limits and retries belong in the runtime

429Honor Retry-After and reduce concurrency pressure.
408 or 504Use async endpoints, longer timeout, or smaller acquisition scope.
500 classRetry with exponential backoff and request identifiers.
400 classFix schema, URL, permissions, credits, or feature access instead of retrying blindly.
Dead letterPersist the failed request and operator reason for later replay.

Cost model

Cost lives across the whole pipeline

Before acquisition

  • Map and filter candidate URLs.
  • Use include/exclude rules and depth limits.
  • Choose cache freshness deliberately.
  • Set concurrency and timeouts intentionally.

After acquisition

  • Dedupe by normalized URL and content hash.
  • Embed changed chunks, not every chunk.
  • Track cost by source and useful output.
  • Archive raw evidence separately from hot retrieval data.

Strategy

Build, buy, or blend based on the real bottleneck

Decision Good signal Risk to validate
Use Firecrawl Rendering, crawling, parsing, and web drift are infrastructure problems. Limits, cost, feature coverage, and vendor dependency.
Build custom Sources require domain-specific browser logic or strict control. Maintenance burden and hidden crawler complexity.
Blend Most sources are standard, but a few need custom extractors. Inconsistent provenance and evaluation across paths.

Database alternatives

MongoDB can own several roles, but not every role must live there

Need Likely store MongoDB position
Application documents and metadata Document database Strong primary store.
Vector and hybrid retrieval MongoDB Vector Search or vector-first store Strong when filters and documents matter.
Large immutable binaries Object storage Store pointers and metadata.
Event streaming Queue or log Useful for state, not a replacement for stream transport.
BI reporting Warehouse or lakehouse Export summarized facts when analytics dominate.

Adoption path

Pilot the full data loop

1. Source setPick one domain and define allowed paths.
2. AcquisitionMap, filter, crawl, and store job/page events.
3. StoragePersist raw evidence, normalized docs, chunks, vectors.
4. RetrievalImplement vector plus text or hybrid search with filters.
5. EvaluationMeasure answer quality, source freshness, failures, and costs.

Anti-patterns

Most bad RAG starts as bad ingestion

Technical mistakes

  • Crawl every page before looking at the URL map.
  • Embed boilerplate, navigation, duplicate pages, and stale chunks.
  • Skip content hashes and source versioning.
  • Treat parseable JSON as verified truth.

Strategic mistakes

  • Compare vendors without a source-quality test set.
  • Ignore refresh costs until the corpus is large.
  • Choose a vector store before defining metadata filters.
  • Forget who owns source policy and retention.

Decision checklist

Ask these before scaling

Question Evidence you want
Can Firecrawl acquire the source reliably?Success rate, page errors, queue time, retry profile.
Is the content useful after cleaning?Human review, duplicate rate, boilerplate rate, extraction accuracy.
Does MongoDB match the query shape?Metadata filters, vector recall, full-text needs, hybrid ranking evidence.
Is freshness affordable?Changed-page rate, cost per refresh, selective re-embedding plan.
Can the system explain itself?Source lineage, citations, job ledger, audit trace, eval history.

Recap

The mental model

Firecrawl acquires

It discovers, scrapes, crawls, searches, interacts, monitors, and converts web surfaces into usable artifacts.

Databases operate

They make ingestion durable, observable, governable, queryable, replayable, and evaluable.

MongoDB fits

It is compelling when document-shaped content, metadata, vectors, filters, hybrid search, and app state belong together.

The goal is not to scrape a page into a prompt. The goal is a controlled data supply chain for AI features.
