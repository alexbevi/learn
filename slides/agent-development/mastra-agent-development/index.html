Mastra for Agent Development

A TypeScript-first path to agents, tools, workflows, memory, RAG, observability, and evals, compared against LangChain, LangGraph, and LangSmith.

55 minutes / 35 slides

Framing

Mastra competes on product shape, not just API syntax

TypeScript-native

Agent code, tool schemas, workflow steps, server routes, and app code can live in the same language and deployment stack.

Integrated primitives

Agents, tools, workflows, memory, RAG, MCP, traces, and scorers are presented as one framework surface.

Builder workflow

Local Studio, typed tools, and first-class observability aim to shorten the loop from idea to production behavior.

Learning goals

By the end, you should be able to explain

How Mastra works

Agents call typed tools, workflows make long-running business processes explicit, and memory/RAG/observability become framework concerns.

How it compares

Mastra overlaps with LangChain agents, LangGraph workflows, and LangSmith observability, but packages those concerns differently.

Where it fits

Mastra is compelling when your application and team already live in TypeScript and you want agent infrastructure inside that stack.

Where to be careful

LangChain's ecosystem maturity, LangGraph's low-level runtime control, and LangSmith's platform depth remain major comparison points.

Running example

Customer operations agent

Incoming request

"Order A-123 is late, the customer is angry, and they want a refund."

Agent responsibilities

  • Classify urgency and risk.
  • Retrieve order, account, and policy context.
  • Draft a response and proposed action.
  • Escalate destructive or high-value actions.
  • Update systems and learn durable preferences.

Common ground

Most agent frameworks converge on the same primitives

Primitive Product question Failure if implicit
Agent loop Who decides the next action? The model wanders without stop conditions.
Tools What can the model do? Side effects are too broad or poorly typed.
Workflow Which steps must be deterministic? Business process hides inside a prompt.
Memory/RAG What context should persist or be retrieved? Everything becomes one vague "memory" bucket.
Observability/evals How do we know the agent is improving? Debugging depends on anecdotes.

Ecosystem map

Mastra is a unified TypeScript framework; LangChain is a layered ecosystem

Mastra

  • Agent class and typed tools.
  • Workflow engine for explicit processes.
  • Memory, RAG, MCP, server, storage, tracing, and scorers.
  • Studio and cloud/platform story around local iteration and deployment.

LangChain ecosystem

  • LangChain: agent harness, models, tools, middleware, integrations.
  • LangGraph: orchestration runtime for stateful, long-running agents.
  • LangSmith: tracing, datasets, evals, prompts, deployment, and debugging.
  • Deep Agents and other harnesses layer on top.

Value proposition

Mastra's pitch: keep agent development in your TypeScript product stack

Fewer boundaries

No separate Python service just to host agent orchestration if your app is already Node, Next.js, Hono, Express, or serverless TypeScript.

Typed contracts

Zod-backed inputs and outputs make tool and workflow boundaries inspectable by both the model and the compiler.

Integrated shipping path

Mastra treats local dev, server routes, storage, observability, and evals as part of the framework rather than an afterthought.

Project shape

The core Mastra unit is a registry of agents, workflows, storage, and telemetry

import { Mastra } from "@mastra/core/mastra";
import { LibSQLStore } from "@mastra/libsql";
import { customerOpsAgent } from "./agents/customer-ops";
import { ticketWorkflow } from "./workflows/ticket-workflow";

export const mastra = new Mastra({
  agents: { customerOpsAgent },
  workflows: { ticketWorkflow },
  storage: new LibSQLStore({
    url: "file:./mastra.db",
  }),
});

The registry pattern matters: it is how local Studio, server APIs, workflows, memory, and observability discover the same application primitives.

Agent

Start with the customer operations agent

import { Agent } from "@mastra/core/agent";
import { createTool } from "@mastra/core/tools";
import { z } from "zod";

export const getOrder = createTool({
  id: "get-order",
  description: "Fetch order, shipment, and account context by order id.",
  inputSchema: z.object({ orderId: z.string() }),
  outputSchema: z.object({
    orderId: z.string(),
    status: z.enum(["processing", "shipped", "delayed", "delivered"]),
    customerTier: z.enum(["standard", "plus", "enterprise"]),
    refundEligible: z.boolean(),
  }),
  execute: async ({ context }) => lookupOrder(context.orderId),
});

export const customerOpsAgent = new Agent({
  id: "customer-ops",
  name: "Customer Operations Agent",
  instructions: "Triage support tickets, cite policy, and avoid irreversible actions without approval.",
  model: "openai/gpt-5-mini",
  tools: { getOrder },
});

Tool design

Typed tools are the agent's action boundary

Mastra

Tools are created with `createTool`, explicit `inputSchema`, optional `outputSchema`, and an `execute` function.

LangChain

JavaScript agents use `tool(fn, { name, description, schema })`, also commonly with Zod schemas.

The shared lesson is framework-independent: tools should be narrow, typed, authorized, idempotent, and named in the language of the business.

Invocation

The first useful version answers with tool-backed context

const response = await customerOpsAgent.generate(
  "Customer says order A-123 is late and wants a refund.",
  {
    threadId: "ticket-9482",
    resourceId: "customer-c_123",
  },
);

console.log(response.text);

Mastra concept

`threadId` scopes the conversation; `resourceId` scopes the user or owning resource for memory and history.

LangGraph comparison

LangGraph uses thread-scoped checkpoints; LangChain agents pass `thread_id` through runtime config when persistence is enabled.

RAG

Add product and policy retrieval as a tool

import { openai } from "@ai-sdk/openai";
import { createVectorQueryTool } from "@mastra/rag";

export const policySearch = createVectorQueryTool({
  id: "policy-search",
  description: "Search refund, escalation, and shipping policy documents.",
  vectorStoreName: "pgVector",
  indexName: "support_policies",
  model: openai.embedding("text-embedding-3-small"),
  enableFilter: true,
});

export const customerOpsAgent = new Agent({
  id: "customer-ops",
  name: "Customer Operations Agent",
  instructions: "Use policySearch before quoting policy. Cite the policy section.",
  model: "openai/gpt-5-mini",
  tools: { getOrder, policySearch },
});

RAG comparison

Mastra makes retrieval a first-class agent tool

Concern Mastra LangChain ecosystem
Tool integration `createVectorQueryTool` returns a model-callable tool. Retrievers, tools, vector stores, chains, and agent middleware compose flexibly.
TypeScript DX Primary value proposition; app and agent code share a stack. JS support exists, but much ecosystem mindshare and examples remain Python-heavy.
Depth Integrated path for common RAG needs. Huge integration surface and mature retrieval patterns.

Memory

Memory is not just conversation history

Thread history

What happened in this support ticket conversation?

Working memory

What stable details about this customer should the agent keep in mind?

Semantic recall

What prior conversations or facts are relevant to this new request?

Mastra's memory story is a major part of its product pitch, not an external notebook you bolt on later.

Memory code

Configure memory explicitly for persistence and recall

import { Memory } from "@mastra/memory";
import { LibSQLStore, LibSQLVector } from "@mastra/libsql";

const memory = new Memory({
  storage: new LibSQLStore({ url: "file:./mastra.db" }),
  vector: new LibSQLVector({ url: "file:./mastra.db" }),
  options: {
    workingMemory: { enabled: true },
    semanticRecall: true,
  },
});

export const customerOpsAgent = new Agent({
  id: "customer-ops",
  name: "Customer Operations Agent",
  instructions: "Remember durable customer preferences only when they are explicit.",
  model: "openai/gpt-5-mini",
  tools: { getOrder, policySearch },
  memory,
});

The exact storage/vector adapter changes by environment; the design point is explicit persistence and recall.

Memory comparison

Mastra and LangGraph split memory in similar conceptual ways

Memory type Mastra framing LangGraph framing
Short-term Threads, resources, conversation history, working memory. Thread-scoped state persisted by checkpoints.
Long-term Semantic recall and configured storage/vector backends. Stores namespaced by user/app, searched separately from checkpoints.
Product risk Bad memories become product behavior. Same: durable memory needs write policy, correction, deletion, and evals.

Workflows

Use workflows when the business process should not be improvised

Agent loop

Good when the model should decide which tool to call next under broad instructions.

Workflow

Good when triage, retrieval, drafting, approval, and side effects need auditable order.

This is Mastra's closest conceptual overlap with LangGraph: explicit execution graphs for work that needs control.

Workflow code

Promote the support process into explicit steps

import { createStep, createWorkflow } from "@mastra/core/workflows";
import { z } from "zod";

const classifiedTicketSchema = z.object({
  ticketId: z.string(),
  orderId: z.string(),
  risk: z.enum(["low", "high"]),
});

const draftSchema = z.object({
  draft: z.string(),
  risk: z.enum(["low", "high"]),
});

const classifyTicket = createStep({
  id: "classify-ticket",
  inputSchema: z.object({ ticketId: z.string(), message: z.string() }),
  outputSchema: classifiedTicketSchema,
  execute: async ({ inputData }) => classify(inputData),
});

const draftResponse = createStep({
  id: "draft-response",
  inputSchema: classifiedTicketSchema,
  outputSchema: draftSchema,
  execute: async ({ inputData }) => {
    const result = await customerOpsAgent.generate(
      `Draft a response for order ${inputData.orderId}. Risk: ${inputData.risk}`,
    );
    return { draft: result.text, risk: inputData.risk };
  },
});

export const ticketWorkflow = createWorkflow({
  id: "ticket-workflow",
  inputSchema: z.object({ ticketId: z.string(), message: z.string() }),
}).then(classifyTicket).then(draftResponse).commit();

Workflow comparison

Mastra workflows and LangGraph both make control flow explicit

Dimension Mastra workflows LangGraph
Developer feel Step-oriented TypeScript workflow API with schemas and composition helpers. State graph runtime with nodes, edges, reducers, checkpointers, and stores.
Best fit TypeScript product teams wanting integrated agent + workflow + server primitives. Teams needing low-level runtime control for long-running, stateful agents.
Abstraction level Product-framework oriented. Runtime-oriented and deliberately low-level.

Human approval

Escalation belongs at the side-effect boundary

Trigger

Refund above threshold, enterprise account, legal/compliance concern, account cancellation, or model uncertainty.

Runtime behavior

Suspend, show context to a human, accept approve/edit/reject, then resume with the decision recorded.

Whether implemented with Mastra workflow suspension or LangGraph interrupts, the durable pause is the important architectural primitive.

MCP

MCP turns tools into portable capabilities

import { MCPServer } from "@mastra/mcp";

export const supportMcp = new MCPServer({
  name: "support-ops",
  version: "1.0.0",
  tools: {
    getOrder,
    policySearch,
  },
  agents: {
    customerOpsAgent,
  },
});

Mastra angle

Expose existing Mastra tools, agents, and workflows through an MCP server.

LangChain angle

MCP is ecosystem-wide plumbing; the differentiator is how smoothly a framework maps local primitives to MCP capabilities.

Observability

Agents need traces, not just logs

Model calls

Prompt, output, latency, tokens, provider errors.

Tool calls

Inputs, outputs, validation, authorization, failures.

Workflow steps

State transitions, retries, pauses, resumes.

Memory/RAG

Retrieved context, source quality, recall behavior.

Mastra stores traces and can export to observability platforms; LangSmith specializes deeply in this layer.

Evals

Scorers turn quality into a regression surface

Correctness

Did the agent identify refund eligibility and customer risk correctly?

Grounding

Did the response cite retrieved policy and avoid unsupported claims?

Action safety

Did the agent avoid destructive actions without approval?

Mastra's scorer/eval story competes with the same operational need LangSmith serves: make agent behavior measurable.

Differentiation

Where Mastra feels different

Language center

Mastra starts from JavaScript/TypeScript product teams. LangChain started broader and Python remains highly influential.

Framework packaging

Mastra bundles adjacent needs into one framework surface. LangChain separates harness, runtime, and platform into related products.

Local builder experience

Mastra Studio is positioned as a natural dev companion to registered agents and workflows.

Where LangChain wins

The incumbent ecosystem matters

Integration breadth

LangChain has years of model, retriever, vector store, middleware, and community integration surface.

LangGraph control

For complex state machines, reducers, checkpointers, stores, time travel, and graph-level control, LangGraph is purpose-built.

LangSmith depth

Tracing, eval datasets, production monitoring, debugging, and prompt workflows are mature platform capabilities.

Where Mastra wins

The TypeScript product-stack argument is strong

App-adjacent agents

Your agent can live beside Next.js, Express, Hono, server routes, product types, and deployment code.

Compiler-backed contracts

Tool and workflow schema mistakes surface closer to development, not just inside model behavior.

Unified learning curve

One framework story for agents, workflows, memory, RAG, MCP, storage, observability, and evals is easier to adopt.

Decision guide

Choose based on system shape, not hype

If your team needs... Mastra is attractive because... LangChain ecosystem is attractive because...
TypeScript-first app integration It is the center of gravity. JS support exists, but ecosystem examples can be more mixed.
Low-level graph runtime control Workflows cover many explicit-process needs. LangGraph is designed specifically for this layer.
Observability/eval platform depth Tracing and scorers are integrated into the framework. LangSmith is a specialized platform with strong ecosystem adoption.
Broad integrations and hiring familiarity Growing quickly, especially for TypeScript teams. Mature incumbent with large community footprint.

Architecture synthesis

A production customer operations agent has the same bones in either stack

Control plane

Agent loop for flexible reasoning; workflow graph for predictable business process.

Data plane

App database for truth, vector index for policy/document evidence, memory store for durable user context.

Quality plane

Traces, evals, feedback, datasets, and release gates that keep behavior from drifting silently.

Implementation path

How I would build the example in Mastra

Iteration Capability Mastra primitive
1 Draft support responses from order context. Agent + typed tools.
2 Ground answers in refund/shipping policy. RAG vector query tool.
3 Remember explicit customer preferences. Memory with storage and semantic recall.
4 Make triage, approval, and updates auditable. Workflow steps and suspension/resume.
5 Improve quality release over release. Traces, scorers, and eval datasets.

Anti-patterns

What not to learn from framework demos

Do not expose giant tools

`runSql` and `callInternalApi` are not product-safe tool boundaries.

Do not let memory write itself

Durable memory needs explicit write policy, review, correction, and deletion.

Do not use agents for fixed logic

If the path is deterministic, encode it as workflow automation and call the model only where judgment is useful.

Mental model

Mastra is best understood as an agent application framework

It is not just a LangChain clone

Its strongest claim is that agent primitives should feel native to TypeScript application development.

It is not magic infrastructure

You still need good tool boundaries, state models, memory policy, side-effect controls, and eval loops.
