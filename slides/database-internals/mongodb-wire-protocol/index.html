MongoDB's Wire Protocol

Legacy opcodes, OP_MSG, OP_COMPRESSED, BSON command transport, and why a standardized client/server protocol enabled MongoDB drivers to talk to compatible systems such as DocumentDB, FerretDB, Cosmos DB, and Percona Server.

90 minutes / 48 slides

Learning Goals

What you should be able to reason about

Packet mechanics

Read a MongoDB wire message from header to BSON payload, including request ids, response ids, opcodes, sections, and compression wrappers.

Protocol evolution

Explain why legacy CRUD opcodes gave way to command documents, OP_QUERY command transport, and finally OP_MSG.

Interoperability

Evaluate what “MongoDB-compatible” means for drivers, proxies, emulators, alternative backends, feature negotiation, and correctness gaps.

Anchor

The wire protocol is the database's language boundary

Application codeCalls driver APIs such as insertOne(), find(), and aggregate().
MongoDB driverBuilds BSON command documents, manages topology, pools sockets, retries, sessions, and cursors.

Wire protocol

Stable message contract over TCP Standard header, opcode-specific body, BSON serialization, optional compression, request/response correlation, and negotiated features.
BSON OP_MSG OP_COMPRESSED hello maxWireVersion
MongoDB serverParses wire messages, authorizes commands, executes against storage and query engines.
Compatible serverImplements enough of the same wire contract to satisfy existing drivers.

Protocol Contract

A driver does not send JavaScript methods; it sends BSON commands

API surface

collection.find({ status: "paid" }) is a language helper, not the protocol itself.

Command document

The driver serializes a command document such as { find: "orders", filter: ... }.

Wire frame

The command document is placed in an opcode-specific binary message and sent across a socket.

Byte Layout

Every message starts with the same 16-byte header

messageLength int32, total message bytes including this header.
requestID int32 chosen by the sender so replies can be correlated.
responseTo int32, usually zero for requests and request id for replies.
opCode int32 identifying OP_MSG, OP_COMPRESSED, or legacy operation format.
Message body Opcode-specific fields followed by BSON documents, document sequences, cursor ids, flags, or compressed bytes.

BSON

BSON gives the wire protocol typed command documents

Binary documentLength-prefixed, little-endian, null-terminated keys, typed element values.
Command shapeCommands, replies, filters, updates, write concern, sessions, and read concern travel as BSON fields.
Native typesObjectId, Date, Decimal128, binary, regex, timestamp, arrays, embedded documents, and numeric variants.
Driver boundaryEach driver maps host-language types to BSON and back.
Compatibility burdenAlternative servers must parse BSON precisely before they can emulate commands correctly.
Not JSONJSON-like syntax is user-facing notation; the socket carries BSON bytes.

BSON Example

A find command becomes a BSON document inside OP_MSG

// Driver API
await db.collection("orders").find({
  status: "paid",
  totalCents: { $gte: 5000 }
}).sort({ createdAt: -1 }).limit(25).toArray()

// Command document before BSON serialization
{
  find: "orders",
  filter: { status: "paid", totalCents: { $gte: 5000 } },
  sort: { createdAt: -1 },
  limit: 25,
  lsid: { id: UUID("...") },
  $db: "shop"
}

Opcode Timeline

The protocol moved from operation opcodes to command transport

1Legacy CRUDOP_INSERT, OP_QUERY, OP_UPDATE, OP_DELETE, OP_GET_MORE, OP_KILL_CURSORS.
2CommandsServer features increasingly exposed through command documents against $cmd.
3OP_QUERY bridgeDrivers sent commands through a query-shaped message for years.
4OP_MSGA command-native envelope with flags, sections, checksums, and better extensibility.
5OP_COMPRESSEDA wrapper that compresses another wire message after negotiation.
6Modern baselineMongoDB 5.1 removed support for legacy opcodes except OP_MSG and OP_COMPRESSED.

Legacy Opcodes

The original opcodes encoded database operations directly

Opcode Intent Body flavor Modern replacement
OP_INSERT Insert one or more documents into a namespace. Flags, full namespace string, BSON document list. insert command over OP_MSG.
OP_QUERY Query collection or send command via $cmd. Flags, namespace, skip, return count, query BSON. find, hello, and other commands over OP_MSG.
OP_UPDATE / OP_DELETE Mutate or delete documents matching a selector. Namespace plus selector and update/delete flags. update and delete commands.
OP_GET_MORE / OP_KILL_CURSORS Continue or kill server cursors. Cursor id fields outside a command document. getMore and killCursors commands.

Why Legacy Hurt

Feature growth did not fit operation-specific message bodies

Feature sprawl

Sessions, transactions, read concern, write concern, collation, API versioning, and retryability all fit naturally as command fields.

Inconsistent envelopes

Each legacy opcode had its own body layout, making new cross-cutting fields harder to standardize.

Command universality

A single command document model lets new server features ship without inventing a new top-level opcode.

OP_QUERY Command Bridge

Before OP_MSG, commands rode through $cmd

// Conceptual legacy command transport
OP_QUERY {
  flags: 0,
  fullCollectionName: "shop.$cmd",
  numberToSkip: 0,
  numberToReturn: -1,
  query: {
    find: "orders",
    filter: { status: "paid" },
    $readPreference: { mode: "primary" }
  }
}

This compatibility bridge mattered historically, but it was still a query-shaped envelope pretending to be command transport.

OP_MSG

OP_MSG is the modern command-native envelope

Header16-byte standard message header with opcode 2013.
flagBitsint32 for checksum, more-to-come, exhaust allowed, and future flags.
Section 0Kind 0: one BSON body document containing command fields.
Section 1Kind 1: optional document sequence for repeated documents.
Optional checksumCRC-32C may follow sections when the checksum flag is set.

Sections

OP_MSG separates command body from bulk document sequences

Kind 0 body

Exactly one BSON command body. It contains fields such as insert, $db, lsid, txnNumber, and options.

Kind 1 document sequence

A typed sequence of BSON documents with an identifier such as documents, updates, or deletes.

OP_MSG {
  flagBits: 0,
  sections: [
    { kind: 0, body: { insert: "orders", ordered: true, $db: "shop" } },
    { kind: 1, identifier: "documents", documents: [
      { _id: ObjectId("..."), status: "created" },
      { _id: ObjectId("..."), status: "paid" }
    ] }
  ]
}

Why Section 1 Matters

Bulk writes avoid nesting every document inside one huge command BSON

Naive body-only shape

Everything nested The command body would contain a massive array of documents, forcing extra structure and size pressure inside one BSON object.

OP_MSG section 1

Command body plus document stream Options stay in the body; repeated documents travel as an adjacent sequence with a named identifier.

Driver/server benefit

Cleaner batching boundary Drivers can split batches by max message size and max BSON object size while preserving command semantics.

Replies

Replies are also command documents

Success

Most replies contain ok: 1, command-specific fields, cursor state, or write outcome fields.

Failure

Errors travel as BSON fields such as ok: 0, errmsg, code, codeName, and optional labels.

Correlation

The reply header uses responseTo to point back at the request's requestID.

{
  cursor: {
    id: Long("734734..."),
    ns: "shop.orders",
    firstBatch: [{ _id: ObjectId("..."), status: "paid" }]
  },
  ok: 1
}

Handshake

The first conversation negotiates topology and capabilities

1Open socketDriver connects to seed host and starts monitoring.
2Send helloClient metadata, compression candidates, API version, and topology context.
3Read replyServer reports role, wire versions, hosts, set name, election id, session timeout.
4Select serverDriver chooses primary, secondary, mongos, or standalone based on read/write operation.
5AuthenticateSASL or other auth exchange uses wire messages too.
6Use featuresDriver enables sessions, retryable writes, transactions, compression, and command options only when supported.

Wire Versions

minWireVersion and maxWireVersion are feature gates

Protocol compatibility

Drivers compare their supported wire version range with the server's range before using modern features.

Not just marketing

A compatible service often emulates wire version values so existing drivers will continue past handshake.

Feature caveat

Passing handshake does not guarantee every command, option, error label, transaction behavior, or aggregation feature is implemented.

Compression

OP_COMPRESSED wraps another complete wire message

HeaderStandard header with opcode 2012.
originalOpcodeOpcode of the message after decompression, usually OP_MSG.
uncompressedSizeint32 size of the original opcode body.
compressorIdOne byte: snappy, zlib, zstd, or noop depending on negotiation.
compressedMessageCompressed bytes for the original message body, not a new command model.

Compression Negotiation

Compression is opt-in and per-connection

Client advertises The initial handshake includes a list such as ["zstd", "snappy", "zlib"]. Driver config
Server selects The server response identifies mutually supported compressor names. Capability negotiation
Messages wrap Eligible messages are sent as OP_COMPRESSED with the original opcode recorded in the wrapper. Transport optimization

Parser Example

A proxy or compatible server starts by decoding the header

import struct

HEADER = struct.Struct("<iiii")  # little-endian int32 x 4

def parse_header(buf: bytes) -> dict:
    if len(buf) < HEADER.size:
        raise ValueError("incomplete MongoDB wire header")
    message_length, request_id, response_to, opcode = HEADER.unpack_from(buf)
    if message_length < HEADER.size:
        raise ValueError("invalid message length")
    return {
        "messageLength": message_length,
        "requestID": request_id,
        "responseTo": response_to,
        "opCode": opcode,
        "body": buf[HEADER.size:message_length],
    }

OP_MSG Parser

After the header, parse flags and section kinds

def parse_op_msg(body: bytes, decode_bson):
    flags = int.from_bytes(body[0:4], "little", signed=False)
    pos = 4
    sections = []
    while pos < len(body):
        kind = body[pos]
        pos += 1
        if kind == 0:
            doc, used = decode_bson(body[pos:])
            sections.append(("body", doc))
            pos += used
        elif kind == 1:
            size = int.from_bytes(body[pos:pos + 4], "little", signed=True)
            payload_end = pos + size
            identifier_end = body.index(b"\x00", pos + 4)
            identifier = body[pos + 4:identifier_end].decode()
            docs = []
            pos = identifier_end + 1
            while pos < payload_end:
                doc, used = decode_bson(body[pos:payload_end])
                docs.append(doc)
                pos += used
            sections.append(("sequence", identifier, docs))
        else:
            raise ValueError(f"unknown OP_MSG section kind {kind}")
    return flags, sections

Driver Path

A driver turns API calls into protocol work plus topology state

1Build commandConvert API inputs into command BSON and validate client-side options.
2Select serverUse topology description, read preference, sessions, and writeability.
3Checkout connectionPick pooled socket with known negotiated capabilities.
4Frame messageEncode OP_MSG, optional document sequences, and maybe compression.
5Read replyDecode header, opcode body, BSON reply, cursor ids, and error labels.
6Apply semanticsRetry, throw, return cursor, update session state, or issue getMore.

Cursor Flow

Queries often become multiple wire round trips

find Client sends command; server returns firstBatch plus a cursor id if more results remain. First reply
getMore Driver requests another batch using cursor id, collection name, batch size, and database. Continuation
killCursors Driver closes server cursor when exhausted, abandoned, timed out, or explicitly closed. Cleanup

Sessions And Transactions

Modern distributed semantics ride as command fields

{
  update: "accounts",
  updates: [
    { q: { _id: "A" }, u: { $inc: { balance: -100 } } }
  ],
  lsid: { id: UUID("...") },
  txnNumber: NumberLong(41),
  autocommit: false,
  startTransaction: true,
  readConcern: { level: "snapshot" },
  $db: "bank"
}

This is the key reason command-native transport matters: protocol evolution happens through BSON fields, not new TCP-level verbs.

Why Interop Works

Drivers trust the protocol boundary, not the storage engine

Client expectation

Speak MongoDB wire protocol Handshake succeeds, commands serialize as BSON, replies follow command conventions, errors look familiar.

Compatibility layer

Translate protocol to backend behavior Parse wire messages, implement command semantics, map errors, maintain cursor/session state, and serialize replies.

Backend reality

Storage can be different MongoDB storage engine, PostgreSQL, cloud-native document store, forked server, or custom query service.

Compliance Testing

The hard part is semantic fidelity, not parsing bytes

Handshake testsDriver can discover server type, wire version, compression, sessions, and auth behavior.
Command testsEvery command has option combinations, error cases, and version-specific semantics.
Operator testsQuery and aggregation operators need edge-case equivalence: nulls, arrays, types, collation, regex.
Cursor testsBatching, cursor ids, getMore, timeout, killCursors, exhaust behavior, and error handling.
Write testsWrite concern, bulk write ordering, duplicate keys, upsert results, retry labels, transaction state.
Tool testsDrivers, ORMs, mongosh, migration tools, BI tools, and backups exercise different corners.

Common Gaps

Where MongoDB-compatible systems most often diverge

Area Why it is hard What to test
Aggregation Large operator surface, expression semantics, memory behavior, spill, stage ordering. Pipelines with arrays, nulls, dates, numeric types, sort/group/window stages.
Transactions Requires session, snapshot, retry, write concern, and backend isolation alignment. Multi-statement rollback, transient errors, retry labels, read/write concern.
Indexes MongoDB index semantics include multikey, sparse, partial, TTL, text, geo, unique rules. Plan-dependent behavior and uniqueness edge cases.
Errors Drivers and applications branch on codes, labels, and write result fields. Error codes, duplicate key shape, retryability labels, bulk result details.

Versioning Trap

Wire version, API version, and product version are different things

Wire version

Negotiated low-level protocol feature range from handshake. It gates driver behavior.

MongoDB server version

Product release with command, query, storage, and operational behavior changes.

Compatibility claim

A third-party product may target “MongoDB 4.0 API” or “5.0 compatible” while using a different internal version model.

Security Boundary

Wire compatibility expands the blast radius of driver behavior

Credential semantics

Driver auth flows, TLS expectations, certificate validation, and connection string parsing all matter.

Command authorization

Compatible services need MongoDB-shaped authz semantics or clearly documented differences.

Protocol parsing

Any server that accepts wire traffic must defend against malformed lengths, oversized BSON, decompression bombs, and parser confusion.

Interop Checklist

How to evaluate a MongoDB-compatible technology

1. Driver handshake Confirm supported drivers connect without disabling modern features unexpectedly. Wire compatibility
2. Command inventory Map every application command, operator, index type, aggregation stage, and admin command to support docs. API coverage
3. Semantic tests Run production-shaped fixtures through MongoDB and the compatible system and diff results/errors. Correctness
4. Operational tests Exercise failover, retries, backup, monitoring, scaling, compression, TLS, and migration tooling. Production fit

Migration Implication

The wire protocol lowers switching cost, but does not erase it

Low friction

Existing drivers, connection pools, ORMs, and some tools can often be reused.

Medium friction

Connection strings, TLS, auth mechanisms, indexes, retry settings, and server parameters may need adjustment.

High friction

Unsupported commands, aggregation semantics, transactions, performance behavior, and operational model changes require application work.

Practical Mental Model

Think in three contracts

Wire contract

Can the driver talk? Message header, opcode, BSON, compression, handshake, auth, request/reply correlation.

Command contract

Can the app express work? Commands, options, operators, indexes, sessions, transactions, cursors, errors.

Semantic contract

Does it behave the same? Results, consistency, isolation, performance, limits, operational behavior, failure modes.

Recap

What the wire protocol explains

MongoDB client/server mechanics

Drivers serialize BSON commands into OP_MSG frames, optionally wrapped by OP_COMPRESSED, and correlate replies by header ids.

Protocol evolution

Legacy opcodes encoded operations directly; modern MongoDB uses command documents over a more extensible message envelope.

Interoperability

MongoDB-compatible systems work by implementing the wire, command, and semantic contracts deeply enough for real drivers and applications.

Takeaways

How to reason about MongoDB-compatible systems

  • Wire compatibility means the driver can speak; it does not automatically mean full MongoDB behavior.
  • OP_MSG made command documents the center of protocol evolution.
  • OP_COMPRESSED is a transport wrapper, not a different command model.
  • BSON type semantics are part of compatibility, not an incidental serialization detail.
  • DocumentDB, FerretDB, Cosmos DB, and similar systems exist because the northbound protocol is stable enough to emulate.
  • Production migration requires semantic, operational, and tooling tests, not only successful connection tests.
