MongoDB Driver Specifications: SDAM and Server Selection

How drivers discover topology, monitor health, select hosts for reads and writes, route operations transparently, and preserve resilience while abstracting a distributed database behind a normal client API.

85 minutes / 42 slides

Learning Goals

What you should be able to reason about

State machine

Explain how a driver turns seed hosts and hello replies into a live TopologyDescription.

Routing

Trace how writes, reads, cursors, transactions, and sharded operations are routed to suitable servers.

Resilience knobs

Choose and debug timeouts, read preference, latency windows, max staleness, pools, retries, and monitoring settings.

Why This Exists

A MongoDB client is talking to a moving distributed system

Application expectationCall insertOne(), find(), or aggregate() and get a result or a useful error.
Driver responsibilityHide election, topology, network, cursor, retry, and pool details behind stable APIs.

Driver state machine

TopologyDescription + ServerDescription + selection rules The driver continuously converts uncertain network observations into routing decisions for individual operations.
SDAM server selection read preference heartbeats pools
Distributed realityPrimaries step down, secondaries lag, mongos routers fail, DNS changes, sockets break.
Correctness constraintRoute writes to primaries, reads to eligible servers, and stateful work to pinned hosts.

Spec Map

Three driver specs work together in practice

SDAM

Server Discovery and Monitoring defines how drivers discover servers, classify them, update topology state, and react to errors.

Server Selection

Defines how a driver chooses a server for a read or write from the current topology description.

CMAP and retries

Connection pools, pool clearing, retryable reads/writes, and transactions sit beside SDAM and affect resilience.

Core Model

The driver keeps an eventually fresh local map

Seed list

Initial addresses From URI hosts or SRV records. Seeds are hints, not necessarily the full deployment.

TopologyDescription

Client-side deployment view Topology type, set name, compatible wire version, logical session timeout, and a map of server descriptions.

ServerDescription

Per-host facts Address, server type, RTT, tags, wire versions, set metadata, election id, last write date, error state.

Server Types

Drivers route by classified server type, not by host string

UnknownNo current usable information, failed heartbeat, or invalidated description.
StandaloneSingle mongod not participating in replica set routing.
MongosRouter for a sharded cluster; drivers can choose among multiple routers.
RSPrimaryWritable replica set member for normal writes and primary reads.
RSSecondaryReadable replica set member when read preference allows it.
RSArbiter / RSOther / RSGhostReplica set members that affect topology but are not normal data-bearing targets.

Topology Types

The topology state machine narrows uncertainty over time

UnknownStarting point for multi-host deployments. Monitor seeds and wait for evidence.
SingleDirect connection or single standalone-style target. Selection rules are constrained.
ReplicaSetNoPrimarySet is known, but no current primary is known. Reads may still be possible.
ReplicaSetWithPrimaryPrimary is known and replica set members are tracked from member lists.
Sharded / LoadBalancedDriver routes to mongos routers or a load-balanced service boundary.

Monitoring Loop

Background heartbeats keep the topology map current

1Schedule checkEach known server has a monitor with periodic heartbeat timing.
2Send helloMonitor opens or reuses its monitoring connection and asks for server status.
3Measure RTTHeartbeat round-trip time feeds the latency window used during selection.
4Parse responseClassify server type, set name, hosts, tags, election id, last write date.
5Update topologyAdd/remove hosts, invalidate stale primary, transition topology type.
6Wake waitersOperations blocked in server selection can retry selection from the new state.

Hello Payload

The hello response is the state machine's input

Identity and role

isWritablePrimary, secondary, arbiterOnly, msg: "isdbgrid", setName, and me.

Membership

hosts, passives, arbiters, hidden members, and replica set reconfiguration metadata.

Selection inputs

Wire versions, tags, last write date, election id, set version, logical session timeout, and load-balanced signals.

Seed Discovery

A seed list is only the beginning

1URI seedsDriver starts with one or more addresses from connection string or SRV lookup.
2Unknown descriptionsEach seed begins as a server description of type Unknown.
3First hello repliesDriver learns whether hosts are replica set members, mongos, standalone, or ghosts.
4Member expansionReplica set primary/secondary replies list canonical members to add and monitor.
5Seed cleanupHosts not in the set, wrong set name, or wrong deployment type are removed.
6Stable mapDriver continues monitoring known members as the deployment changes.

Replica Set Rules

The driver can use a set even when the seed list is stale

Members add members

A primary or secondary reports the set's hosts, passives, and arbiters; the driver adds unknown descriptions for new addresses.

Membership removes hosts

Hosts no longer reported by the set are removed and no longer monitored.

Wrong type is rejected

A standalone in a multi-host seed list is removed instead of collapsing the deployment into Single.

Stale Primary

Election metadata prevents routing to an old primary

New primary discovered A heartbeat observes an RSPrimary with newer setVersion or electionId. Topology update
Old primary invalidated Any previous primary is replaced with an Unknown server description and checked again soon. Correctness guard
Writes stop targeting it Server selection no longer sees the old primary as eligible for writes. Failure containment

Monitoring Modes

Drivers may monitor differently, but must expose equivalent behavior

Mode How topology changes arrive Selection implication
Multi-threaded / async Background monitors update topology while application operations run. Waiting operations are unblocked as soon as a suitable server appears.
Single-threaded Monitoring may be coupled to server selection attempts. serverSelectionTryOnce can control whether one scan is attempted before failing.
Streaming hello Some modern monitors can wait for topology changes instead of only polling periodically. Drivers can react faster to changes while still respecting heartbeat constraints.

Selection Pipeline

Server selection turns topology state into one operation target

1Operation typeWrite, read, command helper, transaction start, cursor continuation.
2Topology filterSingle, replica set, sharded, load-balanced each apply different rules.
3Mode filterRead preference mode or write requirement narrows candidate servers.
4Staleness/tagsMax staleness and tag sets filter secondaries when configured.
5Latency windowKeep servers within fastest RTT plus localThresholdMS.
6SelectChoose suitable server, often with randomization or operation-count weighting.

Writes

Normal writes require a writable target

Replica set

Writes target the known RSPrimary. If no primary is known, selection waits until timeout or a primary appears.

Sharded cluster

Writes target an eligible mongos; the router owns shard targeting inside the cluster.

Load balanced

Driver selects the load-balanced endpoint and lets the service route behind the boundary.

Read Constraints

Staleness and tags refine secondary eligibility

Mode first

Primary / secondary / preferred / nearest Start by deciding which classes of servers may be candidates.

Max staleness

Replication lag budget Filter secondaries that are too far behind based on last write dates and heartbeat timing.

Tag sets

Data center or workload affinity Apply ordered tag sets after staleness; first tag set with matches wins.

Latency Window

localThresholdMS prevents always choosing the absolute fastest ping

Fastest RTT

Use heartbeat round-trip time estimates from server descriptions.

Window

Eligible servers are those whose RTT is at most fastest RTT plus localThresholdMS.

Effect

Spread load across roughly local servers while avoiding obviously distant or unhealthy targets.

Special Routing

Some operations must stay pinned to their original host

getMoreMust go to the server that owns the cursor id and returned the previous batch.
killCursorsTargets the server that owns the cursor being killed.
Transactions on mongosAfter the first sharded transaction operation selects a mongos, later operations are pinned.
Load-balanced cursorsCan require connection or service-id pinning depending on operation type.
Change streamsLong-lived cursor behavior makes resumability and server errors important.
Retry pathsRetries may intentionally avoid a recently failed address when the spec requires deprioritization.

Timeout Budget

Several timeouts control different parts of the path

1Server selectionserverSelectionTimeoutMS or remaining timeoutMS.
2Pool checkoutWait for an available connection or create one within pool constraints.
3ConnectconnectTimeoutMS bounds TCP/TLS connection establishment.
4Socket read/writesocketTimeoutMS or context deadline bounds I/O.
5Server workmaxTimeMS can limit server-side execution for supported commands.
6Write concernwTimeoutMS bounds replication acknowledgement waiting.

Server Selection Timeout

serverSelectionTimeoutMS is the wait for a suitable server

No suitable server The topology has no primary for a write, no matching secondary for a read preference, or all candidates are unknown. Selection waits
Topology changes Background monitoring may discover a suitable server before timeout and wake the operation. Retry selection
Timeout expires The driver raises a server selection error with the current topology description and underlying server errors. Actionable diagnosis

CSOT

timeoutMS gives one client-side operation budget

What it covers

Client-side operation timeout can include server selection, pool checkout, connection establishment, command execution, and retries.

Why it matters

It prevents a stack of independent timeouts from producing unexpectedly long end-to-end latency.

Caveat

Driver support and precedence vary by language and version; know your driver's current CSOT behavior.

More Knobs

Connection and pool settings decide whether selected servers are usable

connectTimeoutMSBounds opening a TCP/TLS connection to a selected or monitored host.
socketTimeoutMSBounds socket reads/writes in drivers that expose it independently.
maxPoolSizeCaps concurrent checked-out connections per server.
minPoolSizeMaintains warm connections at the cost of baseline resources.
maxConnectingLimits concurrent connection establishment after bursts or pool clears.
retryReads / retryWritesControls whether eligible operations are retried after retryable failures.

Failure Handling

Network errors mutate topology state, not only the current operation

1Operation failsSocket error, timeout, not-primary, shutdown, or handshake failure.
2Classify errorDriver applies SDAM error handling rules for server type and topology.
3Mark unknownServerDescription may be replaced with Unknown and error stored.
4Clear poolSome errors clear or pause a connection pool generation.
5Request checkMonitor is asked to heartbeat soon instead of waiting for the next period.
6Reselect/retryEligible retryable operations select again from updated topology.

Election Timeline

The driver preserves correctness during primary failover

T0: primary known Topology is ReplicaSetWithPrimary; writes target member A. Normal routing
T1: A steps down Operation errors or heartbeat observes A is no longer writable; A becomes Unknown or secondary. Invalidate
T2: no primary window Writes wait in server selection while monitors scan; secondary reads may continue if configured. ReplicaSetNoPrimary
T3: B elected Heartbeat observes B as primary; old primary is stale-checked; waiting writes wake and target B. Recovery

Retryability

Retries depend on selection, labels, sessions, and operation type

Retryable reads

Eligible read operations can reselect a server after retryable network-style failures.

Retryable writes

Single-document writes and certain write operations use sessions and transaction numbers to avoid duplicate effects when supported.

Not magic

Retries cannot make non-idempotent operations safe unless the server and driver protocol semantics support it.

Sharded Clusters

Drivers select routers; routers select shards

Driver

Select eligible mongos Monitor router availability and RTT, then choose a router within the latency window.

mongos

Own shard targeting Parse command, consult cluster metadata, route to shards, merge results when needed.

Shards

Replica sets behind routers Driver normally does not select shard replica set members directly for sharded operations.

Load Balanced

Load-balanced mode changes what the driver is allowed to know

One service boundary

The driver treats the endpoint as a load-balanced topology, not as a discoverable replica set.

Service IDs

Responses can identify backing services for pinning and pool behavior.

Operational implication

External load balancing shifts routing and health semantics away from direct host discovery.

Diagnostics

A server selection error is a topology snapshot

MongoServerSelectionError: server selection timed out after 30000 ms
TopologyDescription {
  type: ReplicaSetNoPrimary,
  setName: "atlas-shard-0",
  servers: {
    "a:27017": { type: Unknown, error: "connect ETIMEDOUT" },
    "b:27017": { type: RSSecondary, roundTripTime: 18ms },
    "c:27017": { type: RSSecondary, roundTripTime: 21ms }
  },
  heartbeatFrequencyMS: 10000,
  localThresholdMS: 15
}

Reading The Error

Ask which filter removed all candidates

Topology filter Is the deployment known? Is there a primary? Are hosts reachable? Is the set name correct? SDAM issue
Read policy filter Did read preference, tag sets, or max staleness exclude otherwise healthy servers? Selection issue
Connection filter Was a server selected but connection checkout, DNS, TLS, auth, or socket I/O failed? Pool/network issue

Application-Level Routing

Use read preference intentionally, not as a generic scale knob

// Node.js: analytics read from tagged secondaries
const analytics = client
  .db("shop")
  .collection("orders")
  .withReadPreference({
    mode: "secondary",
    tags: [{ workload: "analytics", region: "us-east" }],
    maxStalenessSeconds: 120
  });

const rows = await analytics
  .find({ status: "paid" })
  .sort({ createdAt: -1 })
  .limit(500)
  .toArray();
// Writes still select the primary
await client
  .db("shop")
  .collection("orders")
  .insertOne({
    customerId: "C-1842",
    status: "created",
    createdAt: new Date()
  });

// Driver routes by operation type,
// not by the collection's last selected host.

Observability

Drivers expose events for the hidden state machine

SDAM events

Topology opening/changed/closed, server opening/changed/closed, heartbeat started/succeeded/failed.

Selection events

Server selection started, succeeded, failed, waiting, and selector details.

Pool events

Pool created, connection created/ready/checked out, wait queue, pool cleared.

Command events

Command started/succeeded/failed with operation ids and durations.

Resilience Pattern

Let the driver own topology; make app retries bounded and observable

Use one long-lived client

A single client owns topology monitoring and pools. Recreating clients defeats the state machine.

Set explicit budgets

Use operation-level deadlines or timeoutMS and choose selection timeouts aligned with user latency.

Log topology errors

Server selection failures contain the current deployment view; capture it before turning everything into a generic 500.

Anti-Patterns

Common ways to fight the driver

One client per requestDestroys pool reuse, monitoring state, and normal failover behavior.
Huge selection timeoutTurns topology problems into long user waits.
Secondary reads everywhereCan create stale reads, inconsistent UX, and harder debugging.
Ignoring tagsCross-region reads happen when read preference is broad and latency window allows it.
Only watching command latencySelection, pool checkout, and connect time can dominate before the command starts.
Retrying above the driver blindlyCan amplify outages if app retries ignore driver retry labels and time budgets.

Checklist

Production checklist for driver routing resilience

1. Topology fit Use the right URI scheme, set name, load-balanced mode, TLS, and direct connection settings. Discovery
2. Routing policy Define read preference, tags, max staleness, and latency window per workload, not globally by accident. Selection
3. Time budgets Align timeoutMS, serverSelectionTimeoutMS, connect, socket, and server-side timeouts. Failure behavior
4. Pool capacity Size pools and connection creation limits from concurrency and latency, not from maximum traffic hopes. Resources

Mental Model

What the driver does for you

Observe

Continuously monitor Heartbeat each known server and maintain topology descriptions with RTT, roles, tags, and errors.

Decide

Select suitable servers Apply operation type, topology, read preference, staleness, tags, latency, and pinning constraints.

Recover

Update state after failure Invalidate stale descriptions, clear pools when required, retry eligible operations, and wake waiting work.

Recap

The hidden state machine is the product

Discovery

SDAM turns a seed list and hello replies into a live topology description.

Selection

Server selection chooses an eligible host per operation using topology, read preference, latency, tags, and staleness.

Resilience

Timeouts, pools, retries, and error handling determine whether failures are brief routing events or user-visible outages.

Takeaways

How to think about driver behavior

  • The driver is not a dumb socket wrapper; it is a topology-aware distributed systems client.
  • Writes route to known writable targets; reads route according to explicit read preference policy.
  • serverSelectionTimeoutMS means “wait for a suitable server,” not “run the database command for this long.”
  • localThresholdMS, tags, and max staleness shape where reads go after basic eligibility.
  • Server selection errors are often the best diagnostic artifact because they include the driver's current view of the deployment.
  • Use one long-lived client, explicit time budgets, and driver events to make failure behavior understandable.
