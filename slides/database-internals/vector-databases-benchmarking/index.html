Vector Databases Benchmarking

A deep technical guide to evaluating vector databases with a Python support-ticket benchmark: MongoDB Atlas Vector Search, Voyage embeddings and reranking, and the usual vector-first alternatives.

80 minutes / 45 slides / Last updated 2026-05-27

Learning goals

Compare systems through evidence, not positioning

Mechanics

Understand embeddings, ANN indexes, exact search, filters, hybrid search, reranking, and update paths.

Benchmarking

Design a fair Python harness with fixed corpus, vectors, labels, queries, filters, metrics, and adapters.

Systems

Compare MongoDB Atlas Vector Search with Pinecone, Weaviate, Qdrant, Milvus, Chroma, and pgvector.

Decisions

Use support-ticket retrieval evidence to choose storage, serving, and embedding architecture.

Running workload

Support and customer-ticket retrieval

Corpus

Tickets, comments, resolution notes, product area, severity, tenant, customer tier, timestamps, and internal tags.

Queries

Natural-language symptoms mixed with exact identifiers: invoice ids, error codes, product names, regions, and time windows.

This workload is useful because it punishes pure vector demos. Real support search needs semantics, lexical recall, metadata filters, freshness, and tenant isolation.

Boundary

What a vector database owns

VectorsStore embeddings with ids and metadata.
IndexANN or exact structures for nearest-neighbor search.
FiltersConstrain candidates by tenant, product, date, or status.
ServingLatency, throughput, concurrency, updates, deletes, and scale.

It does not own embedding quality. That is why the benchmark fixes Voyage embeddings and swaps only the database adapter.

Benchmark topology

The adapter is the comparison boundary

Fixed inputsTickets, queries, qrels, filters, and Voyage vectors.
Cached artifactsJSONL corpus, embeddings cache, query vectors, relevance labels.
Python harnessAdapter contract, ingest runner, query runner, reranker, metric calculator.
Store adaptersMongoDB, Pinecone, Weaviate, Qdrant, Milvus, Chroma, pgvector.
OutputsRuns, metrics, traces, errors, latency histograms, cost model.
Decision reportScorecard plus qualitative failure analysis.

Corpus shape

Support tickets need rich metadata

{
  "id": "ticket_9182",
  "tenant_id": "acme",
  "product": "billing",
  "priority": "p1",
  "created_at": "2026-04-19T14:11:00Z",
  "text": "Customer reports invoice doubled after plan change...",
  "resolution": "Proration rule incorrectly applied after renewal.",
  "embedding": [0.013, -0.021, 0.044]
}

The metadata fields are not decoration. They are benchmark dimensions.

Queries

Use queries that look like support work

Semantic symptomCustomer says monthly invoice doubled after renewal.
Exact signalError BILLING-409 appears after plan migration.
Structured constraintTenant acme, product billing, last 90 days.

Pure semantic search can miss exact identifiers; pure lexical search can miss paraphrased symptoms.

Labels

No qrels means no quality benchmark

{
  "query_id": "q_billing_proration_001",
  "query": "invoice doubled after plan change",
  "filters": { "product": "billing", "tenant_id": "acme" },
  "relevant": {
    "ticket_9182": 3,
    "ticket_7741": 2,
    "ticket_6139": 1
  }
}

Use binary labels for recall, or graded labels when nDCG matters.

Cache vectors by model, input type, text hash, and dimension.

Retrieval pipeline

Measure every stage separately

Embed queryVoyage query vector from fixed model.
FilterTenant, product, priority, date range.
RetrieveTop N candidates from one adapter.
RerankOptional Voyage pairwise reranking.
ScoreRecall, nDCG, MRR, latency, errors.

ANN versus exact

Approximate search trades recall for speed

Exact search

Compares against all eligible vectors. Useful for small sets, validation, and ground-truth calibration.

ANN search

Uses an index to avoid exhaustive comparison. Useful at scale, but it can miss relevant candidates.

In benchmark reports, ANN recall loss should be visible rather than hidden behind final answer quality.

Index mental model

HNSW is a navigable neighbor graph

BuildConnect vectors into a layered neighbor graph.
SearchNavigate from entry points toward closer candidates.
TuneTrade index memory/build cost for search recall and latency.
UpdateIncremental changes can behave differently by engine.

Scale mechanics

Index families manage memory, disk, and candidate generation

FamilyPurposeBenchmark pressure
HNSWFast graph-based ANN.Memory footprint, recall/latency tuning, updates.
IVFPartition vectors into coarse clusters.Candidate coverage and cluster probe settings.
Disk-orientedReduce memory pressure by using disk-backed access.p95/p99 latency and warm/cold behavior.
QuantizationCompress vectors or index representation.Memory/cost savings versus recall loss.

Distance metrics

Metric mismatch makes comparisons meaningless

CosineCompares angle. Common for normalized text embeddings.
Dot productCan encode magnitude when vectors are not normalized.
EuclideanDistance in vector space; less common for modern text retrieval setups.

Normalize metric semantics across MongoDB, Pinecone, Weaviate, Qdrant, Milvus, Chroma, and pgvector before scoring.

Chunking

Ticket chunking changes retrieval behavior

UnitStrengthFailure mode
Whole ticketPreserves case context.Long noisy tickets dilute the signal.
CommentFinds precise symptoms and fixes.Can lose full case resolution context.
Resolution noteGreat for known-fix retrieval.Misses unresolved symptoms.
Hybrid fieldsBalances symptoms and outcomes.Requires careful text construction.

Metrics

Use a scorecard, not one number

QualityRecall@50, nDCG@10, MRR@10, precision@k.
Servingp50/p95/p99 latency, QPS, timeout rate.
IngestThroughput, index build time, update lag.
CorrectnessFilter accuracy, tenant isolation, delete visibility.
CostEmbedding, reranking, storage, compute, operations.
OperationsBackup, scaling, observability, schema/index management.

Recall

Recall@k protects the reranker from missing candidates

First-stage target

Use Recall@50 or Recall@100 to measure whether the database can surface the right tickets at all.

Why it matters

If the relevant ticket is absent from candidates, Voyage reranking cannot rescue final quality.

Ranking

nDCG and MRR measure different user pain

nDCG@10

Rewards high-quality graded relevance near the top of the result list.

MRR@10

Rewards getting the first relevant ticket as high as possible.

For support agents, both matter: analysts need the first useful answer quickly, but also need a strong top set.

Latency

Report p95 and concurrency, not only average latency

p50Typical interactive behavior.
p95Most user-visible pain starts here.
p99Operational tail risk.
QPSThroughput under fixed concurrency.

Separate cold index behavior, warmed cache behavior, and steady-state load behavior.

Cost

Vector search cost is not only the database query

EmbeddingDocuments, queries, retries, model upgrades.
RerankingTop N pairs per query.
StorageVectors, metadata, raw text, replicas.
ComputeIndex build, query serving, filters, concurrency.

Python harness

Every store implements the same contract

from typing import Protocol, Sequence

class VectorStore(Protocol):
    name: str

    def recreate(self, *, dim: int, metric: str) -> None: ...
    def upsert(self, records: Sequence[VectorRecord]) -> None: ...
    def query(self, vector: list[float], *, top_k: int, filters: dict) -> list[Hit]: ...
    def delete(self, ids: Sequence[str]) -> None: ...
    def stats(self) -> dict: ...

The harness changes adapters, not benchmark semantics.

Result normalization

Normalize before scoring

@dataclass(frozen=True)
class Hit:
    id: str
    score: float
    rank: int
    metadata: dict
    raw: dict | None = None

def normalize(store_hits: list[dict]) -> list[Hit]:
    return [
        Hit(id=h["id"], score=float(h["score"]), rank=i + 1, metadata=h.get("metadata", {}))
        for i, h in enumerate(store_hits)
    ]

Score direction and distance semantics differ; ranks are safer for cross-store metrics.

MongoDB Atlas Vector Search

MongoDB's differentiator is data co-location

Vector search role

Embed ticket text and query similar tickets through an Atlas Vector Search index.

Platform role

Keep vectors near operational ticket documents, metadata, full-text search, and application queries.

The benchmark should test MongoDB as a document database plus vector search system, not as a detached ANN library.

MongoDB adapter

Store ticket state and embedding together

def upsert(self, records: Sequence[VectorRecord]) -> None:
    ops = [
        ReplaceOne(
            {"_id": r.id},
            {
                "_id": r.id,
                "text": r.text,
                "embedding": r.vector,
                "tenant_id": r.metadata["tenant_id"],
                "product": r.metadata["product"],
                "priority": r.metadata["priority"],
                "created_at": r.metadata["created_at"],
            },
            upsert=True,
        )
        for r in records
    ]
    self.collection.bulk_write(ops, ordered=False)

Pinecone

Pinecone is the managed vector-first reference point

Design centerManaged vector index with explicit dimension and metric setup.
Benchmark focusFiltered search latency, service ergonomics, scale, and cost.
WatchoutOperational data usually lives elsewhere, so joins and source-of-truth updates matter.

Weaviate

Weaviate brings vector, lexical, and object concepts together

IndexesHNSW, flat, dynamic, and compression concepts.
SearchVector, keyword, and hybrid search modes.
Benchmark focusHybrid quality, memory footprint, and filter behavior.

Qdrant

Qdrant makes payload filtering central

PayloadMetadata stored with points and used in filters.
IndexingVector and payload indexes both affect benchmark behavior.
Benchmark focusTenant/product/date filters and quantization tradeoffs.

Comparison matrix

Systems optimize for different boundaries

SystemDesign centerStrong benchmark signalWatchout
MongoDB AtlasOperational documents plus vector/full-text search.Metadata-rich support tickets and hybrid app queries.Must test vector recall like every other store.
PineconeManaged vector-first service.Vector serving ergonomics and filtered ANN latency.Operational data integration lives outside.
WeaviateVector/object/hybrid search platform.Hybrid retrieval and index configuration.Memory/index sizing and tuning matter.
QdrantVector search with payload indexing/filtering.Filtered retrieval and payload-heavy workloads.Filter/index design drives results.

Milvus

Milvus is the index-family and scale benchmark target

Index breadthHNSW, IVF, disk-oriented, and quantization-related options depending on version/config.
Filtered searchSupport-ticket filters need explicit measurement.
Benchmark focusLarge corpus, index build, memory, storage, and operational complexity.

Chroma

Chroma is a useful local baseline

Use it for

Harness development, local experiments, quick corpus checks, and regression tests.

Do not overclaim

Do not present local Chroma results as equivalent to production managed-service behavior.

pgvector

pgvector wins when SQL ownership matters

Data boundaryVectors live with relational customers, tickets, entitlements, and permissions.
Index choicesHNSW and IVFFlat give different build/query tradeoffs.
Benchmark focusSQL filters, query plans, transactions, and operational simplicity.

Hybrid retrieval

Support search needs semantic plus lexical recall

Semantic finds

Paraphrased symptoms, similar resolutions, and conceptual matches.

Lexical finds

Error codes, invoice ids, internal feature names, customer-specific wording, and exact logs.

Benchmark a pure-vector track and a hybrid track so the architecture decision is visible.

Filter selectivity

Filters can dominate vector search behavior

BroadAll tickets across tenants and products.
TenantOnly one customer or organization.
ProductSubset such as billing or auth.
DateRecent tickets or incident window.
PriorityP1/P2 only, often a tiny subset.

Run low-, medium-, and high-selectivity suites. A store can win unfiltered and lose under selective filters.

Freshness

Support tickets mutate after ingestion

New commentEmbedding may need refresh.
Status changeMetadata filter behavior changes.
DeleteResult visibility must disappear.
MergeDuplicate tickets need id and relevance handling.

Measure update lag and deletion visibility as first-class correctness metrics.

Voyage reranking

Reranking is a second-stage experiment

reranked = vo.rerank(
    query=query.text,
    documents=[hit.text for hit in candidates[:50]],
    model="rerank-2",
    top_k=10,
)

final_ids = [candidates[item.index].id for item in reranked.results]

Report candidate Recall@50 before reranking and final nDCG@10 after reranking.

Failure analysis

Look at misses, not just aggregate metrics

False negativeRelevant ticket missing from candidates.
False positiveSemantically close but wrong resolution.
Filter missCorrect ticket excluded or wrong tenant included.
Freshness missUpdated ticket state not reflected.

Tradeoff

MongoDB versus vector-first systems

MongoDB Atlas Vector Search

Compelling when ticket/customer data, metadata, operational updates, and search belong in the same application data platform.

Vector-first systems

Compelling when specialized vector serving, managed vector operations, ANN tuning, or vector scale dominates the architecture.

MongoDB plus Voyage

A strong baseline for MongoDB-centered support systems

VoyageEmbeddings and reranking become fixed retrieval-model components.
Atlas Vector SearchANN retrieval over ticket embeddings stored with documents.
Atlas SearchHybrid search path for error codes, product names, and lexical signals.

When vector-first wins

Move when the benchmark proves vector serving is the bottleneck

ScaleVery large vector corpus and specialized serving needs.
TuningANN knobs materially improve quality/latency.
OperationsTeam wants vector infrastructure owned separately.
CostSpecialized serving wins at target QPS and corpus size.

Smaller alternatives

pgvector and Chroma are not consolation prizes

pgvector

Strong when SQL ownership, transactions, relational filters, and operational simplicity are the main constraints.

Chroma

Strong for local development, small internal tools, and benchmark harness iteration before managed infrastructure.

Decision framework

Choose by workload boundary

Choose MongoDB AtlasTickets already live in MongoDB; filters, updates, full text, and document context matter.
Choose vector-firstVector serving scale, ANN tuning, and managed vector operations dominate.
Choose pgvectorPostgres is the system of record and SQL filters/transactions matter most.
Use Chroma locallyYou need fast local iteration, not production-scale service claims.
Add Voyage rerankingCandidate recall is good but top-10 ordering still disappoints users.
Add hybrid searchError codes, invoice ids, and product names are critical to retrieval.

Checklist

A benchmark worth trusting is reproducible

Version inputsCorpus, queries, qrels, filters, vectors.
Pin settingsModel, dimension, metric, top-k, index config.
Capture outputsRuns, hits, scores, latencies, errors, costs.
Inspect failuresSample misses and explain them before deciding.
