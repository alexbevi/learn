MongoDB's WiredTiger Storage Engine

A deep dive into how MongoDB operations move through the server layer into WiredTiger B-trees, transactions, cache, eviction, journaling, checkpoints, and recovery.

85 minutes / 50 slides

Learning Goals

What you should be able to reason about

Internals

Explain B-trees, pages, update chains, timestamps, cache, eviction, reconciliation, journal, and checkpoint roles.

Operations

Map latency, cache pressure, disk growth, checkpoint stalls, and scan regressions back to engine mechanisms.

Application impact

Predict how schema shape, index count, batching, transactions, and aggregation design change storage-engine work.

Through-Line

We will trace two concrete operations

Persistence path

A multi-document insert into an orders collection with secondary indexes, durable write concern, and crash recovery implications.

Retrieval path

An aggregation pipeline that filters, sorts, groups, and fetches documents through query planning and storage cursors.

The point is not memorizing every internal struct. It is seeing which layer owns which responsibility.

Mental Model

WiredTiger is not just the file format

Access method B-tree tables expose sorted key/value access for collection records and secondary indexes.
Concurrency engine Transactions, snapshots, timestamps, and update visibility let readers and writers overlap.
Memory manager Internal cache, eviction workers, dirty-page thresholds, reconciliation, and history store movement.
Durability engine Write-ahead logging, checkpoints, metadata, turtle file, and restart recovery.
Compression boundary On-disk blocks can be compressed while in-cache collection data is manipulated uncompressed.
Operational surface Metrics expose cache behavior, checkpoint work, transaction pressure, cursors, locks, and I/O symptoms.

Boundary

MongoDB owns semantics; WiredTiger owns storage mechanics

Concern MongoDB server WiredTiger
Command shape CRUD, aggregation, transactions, sessions, read/write concern. Storage cursors, transactions, timestamps, table handles.
Data model Database, collection, document, index, catalog metadata. Tables, row-store keys, pages, update chains, data files.
Query execution Planner, SBE/classic execution, pipeline stages, document materialization. Cursor traversal, page reads, visible-version resolution.
Durability contract Write concern, journaling policy, replication acknowledgement. Commit records, log flush, checkpoint, recovery.

Tables

A MongoDB collection maps to several WiredTiger tables

Collection table

Keys are internal record identifiers. Values are BSON document bodies. Inserts add sorted keys to the collection's B-tree.

Index tables

Each secondary index has its own sorted key space. Compound, multikey, unique, and partial rules affect key generation.

Catalog and metadata

MongoDB tracks collection/index metadata above WiredTiger; WiredTiger metadata tracks files, checkpoints, and handles.

B-Trees

Pages are the unit you keep paying for

B-tree shape

root page
keys + refs
internal page
keys + refs
leaf page
keys + values

Pages split as sorted key ranges grow.

Leaf page

recordId
BSON document for collection table.
index key
Key pattern plus recordId for a secondary index table.
insert
New entries can sit in in-memory insert lists before reconciliation.
update
Existing entries accumulate update structures.

Update chain

newest value transaction id + timestamp decide visibility
older value may still be needed by a snapshot reader
disk image reconciled page version from the data file
history store older durable versions when needed

Row Store

Why B-trees fit MongoDB's access patterns

Sorted navigation

Index range scans are natural B-tree walks. Equality scans narrow to a key range; range queries walk adjacent keys.

Localized mutation

Inserts and updates target leaf pages. If pages exceed configured boundaries, the tree splits and grows.

Separate access paths

Collection and index B-trees let query planning choose between direct record fetch, index traversal, or collection scan.

MVCC

An update chain is the engine's local version history

New write The newest update structure is linked at the head for a key. Fast for current readers
Snapshot reader Visibility checks compare transaction ids and timestamps against the reader's snapshot. Older readers can still see older versions
Reconciliation When a dirty page is written, current values can go to the data store and required old versions to the history store. Version retention has a cost

Timestamps

MongoDB uses logical time to coordinate visibility and stability

Read timestamp

Controls which timestamped updates a transaction can see. Snapshot reads need repeatable visibility.

Oldest timestamp

Marks the point before which versions should be globally visible and no longer needed by readers.

Stable timestamp

Defines what can be checkpoint-durable; recovery rolls back updates newer than the stable boundary when required.

History Store

The history store is not a query feature; it is MVCC infrastructure

Purpose

When an older version must survive beyond the in-memory update chain, reconciliation can write that version to the history store so snapshot readers still have a consistent view.

Operational implication

Long-running reads, pinned timestamps, and heavy updates can increase history retention, cache pressure, and recovery work.

Cache

There are two caches in the path, with different contents

WiredTiger cache

Mutable engine pages In-memory pages, update chains, insert lists, and dirty content.

Primary pressure point for reads, writes, eviction, and checkpoint preparation.

Filesystem cache

Durable page images On-disk page images, usually compressed as stored in data files.

Cold reads may still avoid physical disk if the OS cache is warm.

Application memory

Execution overhead Query stages, aggregation buffers, connections, drivers, and process overhead.

Too much non-cache memory reduces what the OS can use for storage caching.

Eviction

Eviction is the engine's backpressure system

Target Keep cache content below configured targets for total content, dirty content, and update-heavy content.
Workers Eviction server and worker threads find and evict candidate pages across trees.
Application threads When triggers are crossed, user threads can be forced to help evict before they continue work.
Clean page Can be removed from memory because the disk version already matches.
Dirty page Must be reconciled so new data, old data, and tombstones land in the right durable places.
Operational signal Latency spikes often appear when foreground work is pulled into eviction and reconciliation.

Reconciliation

Dirty eviction is where logical versions become page images

Gather page state Merge disk image, insert lists, update chains, tombstones, and visibility rules. Engine state becomes a serializable page
Choose durable content Write visible data to the data store and preserve older versions that active snapshots can still require. History store may grow
Release memory pressure Once the page has a valid durable representation, cache can drop or reduce the in-memory version. Eviction succeeds only after this work

Checkpoints

A checkpoint is a consistent durable base image

Transactionally consistent

Checkpoint runs with a consistent view, coordinates B-tree traversal, block allocation, metadata, and history store checkpointing.

Dirty work reduction

Before the critical checkpoint work, eviction is used to reduce dirty cache content and avoid writing everything in one burst.

Recovery anchor

On restart, WiredTiger begins from the most recent checkpoint and replays the log for committed changes after it.

Journal

The write-ahead log protects changes between checkpoints

Commit log record

Transaction modifications are collected and written as log records that recovery can replay.

Log sequence numbers

WiredTiger uses LSNs to identify log record positions and checkpoint relationships.

Flush and sync

Journal behavior intersects with MongoDB write concern, latency, filesystem semantics, and storage device behavior.

Durability

Commit, journal, and checkpoint are different moments

T0 WT transaction commits Collection and index updates become committed in memory and visible by transaction and timestamp rules.
T1 Log record exists The write-ahead log records modifications between checkpoints so recovery can replay committed changes.
T2 Journal sync boundary Client acknowledgement depends on write concern and journaling behavior, not only on cache mutation.
T3 Checkpoint reconciles Dirty pages are written as a transactionally consistent view of each B-tree and history store.
Crash Recovery starts here WiredTiger opens from the last checkpoint, replays later logs, and rolls back unstable data.
Journal: protects changes since the last checkpoint.
Checkpoint: creates the durable base image.
Recovery: combines both, then enforces timestamp stability.

Persistence Example

The workload: inserting order events

db.orders.insertMany(
  [
    {
      _id: ObjectId(),
      customerId: "C-1842",
      status: "created",
      region: "ca-east",
      totalCents: 4215,
      createdAt: ISODate("2026-05-27T12:00:00Z"),
      items: [{ sku: "A-42", qty: 2 }]
    },
    { customerId: "C-1842", status: "paid", region: "ca-east", totalCents: 4215 }
  ],
  { ordered: true, writeConcern: { w: "majority", j: true } }
)

Insert Path

One batch turns into many engine-level changes

1 Parse batch Validate BSON shape, namespace, ordered behavior, session, transaction, and write concern. Server layer
2 Acquire intent locks Protect catalog and collection metadata while allowing document-level concurrency inside WiredTiger. Concurrency boundary
3 Open WT transaction Attach operations to a snapshot-isolated WiredTiger transaction and a MongoDB write unit. Atomicity boundary
4 Mutate B-trees Insert collection records and every required secondary index key into row-store trees. Write amplification
5 Commit and log Commit update chains in memory and record durable changes in the write-ahead log when journaling applies. Durability boundary
6 Acknowledge Return after configured write concern and journal or replication requirements are satisfied. Client contract
Ordered insert: stops at the first error in the batch.
Unordered insert: can continue independent documents after errors.
Indexes: each extra index adds B-tree mutations and cache pressure.

Batch Semantics

ordered changes failure behavior, not storage fundamentals

Mode Server behavior Storage impact
Ordered Stop processing after the first write error in the batch. Successfully inserted earlier documents still performed collection and index mutations.
Unordered Continue independent inserts after errors where possible. Can improve throughput by avoiding strict sequence dependency, but still creates the same per-document storage work.
Transaction All-or-nothing semantics across the transaction boundary. More versions, locks, history retention, and commit coordination may accumulate before completion.

Write Unit

MongoDB's write unit has to keep collection and indexes consistent

Generate record Assign or validate _id, validate document constraints, and choose the collection record id. Collection table mutation
Generate index keys For every index, derive keys from the document, including multikey expansion and uniqueness checks where applicable. Index table mutations
Commit or roll back Either every participating B-tree mutation commits under the unit, or none of the failed unit becomes visible. Atomic storage boundary

Index Amplification

Each secondary index adds a second write workload

CPU

Build key strings, apply collations, expand array fields, check uniqueness, and compare keys in sorted order.

Cache

Each index B-tree has its own hot pages, splits, dirty content, and eviction candidates.

I/O

More dirty pages means more reconciliation, more checkpoint work, and more log volume.

Read-optimized indexing is also write amplification. That tradeoff is storage-engine real, not just planner-level.

Unique Indexes

Uniqueness turns inserts into reads plus writes

Before insert

MongoDB has to determine whether the generated unique key already exists or conflicts with another uncommitted operation.

After insert

The index B-tree must contain the key mapping to the inserted record id, and the transaction must preserve consistency with the collection table.

A unique secondary index is often the difference between append-like writes and contention-prone writes.

Multi-Document Transaction

Transactions widen the lifetime of versions and resources

More work before commit Updates accumulate under one logical transaction and must fit within memory and server limits.
Longer visibility window Other readers may need older values while the transaction is active or while timestamps are pinned.
Prepared state Distributed transactions can use prepared transactions, which create special conflict and timestamp behavior.

Write Concern

Client acknowledgement layers above engine commit

Layer Question Why latency changes
WiredTiger commit Did the transaction commit inside the engine? CPU, cache, page mutation, and logging work.
Journal Does the configured acknowledgement require journal durability? Log flush and filesystem/device sync path.
Replication Does w require majority or named replica set acknowledgement? Network, secondary apply, oplog, and durable majority commit point.

Crash Recovery

Recovery reconstructs the boundary between stable and unstable work

Open last checkpoint Load the metadata and data-file state from the last complete checkpoint. Known durable base
Replay log Apply committed log records after the checkpoint so changes since the checkpoint are not lost. Journal durability
Rollback to stable Discard timestamped updates newer than the stable point where required by MongoDB replication semantics. Consistency boundary

Retrieval Example

The workload: operational revenue aggregation

db.orders.aggregate([
  { $match: {
      region: "ca-east",
      status: { $in: ["paid", "shipped"] },
      createdAt: { $gte: ISODate("2026-05-01") }
  }},
  { $sort: { customerId: 1, createdAt: -1 } },
  { $group: {
      _id: "$customerId",
      revenue: { $sum: "$totalCents" },
      lastOrderAt: { $first: "$createdAt" }
  }},
  { $sort: { revenue: -1 } },
  { $limit: 50 }
], { allowDiskUse: true })

Aggregation Path

The storage engine sees cursor work, not a pipeline

MongoDB execution layer

Parse pipeline Validate stages and expressions.
Optimize Move filters, coalesce stages, expose indexable predicates.
Plan Select index scan, collection scan, sort, group, join shape.
Execute SBE or classic stages pull batches from storage cursors.
Goal: minimize keys examined, documents fetched, blocking memory, and spills.

WiredTiger retrieval layer

Index B-tree Traverse sorted keys and return record identifiers.
Collection B-tree Fetch BSON records when the plan is not covered.
Cache Use resident pages or read compressed pages when cold.
Snapshot Resolve visible updates from chains, disk image, or history store.
Eviction Keep cache under target while scans and writes compete.
Spill files Blocking stages may write temporary data outside WiredTiger data files.
Storage cost: each missing index or wide fetch becomes page traffic.

Optimizer

Aggregation optimization decides what can reach indexes early

Predicate movement

Filters can be split and moved before projections when dependencies allow, exposing earlier index use.

Stage coalescence

Adjacent stages such as sort and limit can combine, reducing intermediate work and memory.

SBE eligibility

Some stages can run in the slot-based engine when preceding stages are also compatible.

Planner

The planner turns predicates into B-tree traversal choices

Plan shape Storage path Common smell
IXSCAN covered Index B-tree scan returns all required fields from keys. Usually efficient; watch key count and sort direction.
IXSCAN + fetch Index scan returns record ids, then collection B-tree fetches full BSON. Many documents fetched means page traffic and cache churn.
COLLSCAN Collection B-tree scan walks record ranges without selective index narrowing. Often catastrophic when working set exceeds cache.

Covered Query

A covered plan can avoid collection page fetches

db.orders.createIndex({
  region: 1,
  status: 1,
  createdAt: -1,
  customerId: 1,
  totalCents: 1
})

Storage effect

If the pipeline can be satisfied from index keys until a blocking stage, WiredTiger touches fewer collection pages. The tradeoff is a wider and more expensive index to maintain.

Blocking Stages

$sort and $group are memory-shaping stages

Streaming sort

If index order matches the requested sort after predicates, the engine can stream results in order.

Blocking sort

If order is not available from an index, MongoDB must buffer and sort, with possible disk spill.

Grouping

$group aggregates state by key; cardinality drives memory, spill, and downstream latency.

Spills

Aggregation spill is not the same as WiredTiger eviction

Aggregation spill

Execution stages write temporary data when blocking memory requirements exceed limits and disk use is allowed.

WiredTiger eviction

Storage-engine pages are reconciled or discarded to keep the internal cache within limits.

Both can show up as disk I/O, but the remediation is different: plan shape versus cache/write pressure.

Snapshot Reads

Reads pay for old versions when old snapshots are still valid

Current read Usually sees the newest committed visible update and can avoid deep version-chain traversal. Low MVCC cost
Snapshot read May need to walk update chains and history store records to find the version visible to that timestamp. Version-resolution cost
Long operation Can pin timestamps and prevent cleanup of versions that newer work no longer needs. Retention cost

Concurrency

Document-level concurrency lives below intent locks

Intent locks

MongoDB uses collection/database lock modes to protect metadata and coordinate operations.

Engine conflicts

WiredTiger detects write conflicts and visibility conflicts at the storage level; callers retry where appropriate.

Prepared conflicts

Prepared transactions can make readers wait or retry because updates are neither normally visible nor rolled back.

Explain

explain() tells you how much storage work the plan requested

Keys examined

How much index B-tree work the plan performed.

Docs examined

How many collection records were fetched after index traversal or scan.

Spill metrics

For supported stages, execution stats expose whether data spilled and how much temporary data was written.

Profiler

Use multiple instruments because no single one owns the whole path

Instrument Best for Blind spot
Profiler / slow query log Operation shape, keys/docs examined, locks, yields, plan summaries. Does not fully explain storage-device latency.
serverStatus Cache, eviction, checkpoint, transaction, lock counters. Counters require rates and correlation with workload windows.
OS / Atlas metrics Disk latency, queue depth, memory pressure, CPU, filesystem behavior. Cannot distinguish bad query shape from storage pressure without MongoDB metrics.

Application Impact

Schema design changes page and index locality

Document growth

Growing documents can rewrite larger values and create page churn; bounded document shapes are easier for cache locality.

Arrays

Multikey indexes can multiply index keys per document, increasing insert cost and index B-tree size.

Hot keys

Monotonic or low-cardinality patterns can concentrate writes or scans in narrow regions of a tree.

Application Impact

Index design is a storage budget, not only a query feature

Read budget Does the index reduce keys examined, avoid fetches, preserve sort order, and keep hot pages resident? Query latency
Write budget How many extra B-tree mutations, uniqueness probes, page splits, and checkpoint bytes does the index add? Insert/update latency
Cache budget Does the working set of all collection and index pages fit within useful memory? Operational stability

Application Impact

Batching changes overhead shape, but not atomic work per document

Better batching

Fewer network round trips and fewer command-level overheads. More chance for throughput-oriented scheduling.

Worse batching

Large batches can concentrate dirty cache growth, transaction memory, index hot spots, and journal bursts.

Practical rule

Batch for throughput, but observe p95 latency, eviction pressure, journal latency, and secondary apply lag.

Application Impact

Aggregation design either narrows early or pays late

Early $match

Expose selective predicates before expensive transformations so indexes can narrow B-tree traversal.

Sort alignment

Design compound indexes to support the filter and sort prefix where the pipeline needs ordered streams.

Projection discipline

Carry only fields needed by later stages; covered or narrow fetches reduce collection page traffic.

Tuning

Be cautious with knobs that hide design problems

Knob When it helps Risk
WiredTiger cache size Workload truly needs a different split between engine cache and OS/application memory. Too large can starve filesystem cache and other processes; too small increases eviction pressure.
Compression Storage and I/O savings matter more than extra CPU. Can add CPU cost and does not reduce in-cache collection data the same way it reduces disk bytes.
Compaction You need to return reusable internal space to the filesystem after major deletes. Operationally expensive; not a replacement for data lifecycle design.

Failure Modes

Common symptoms and first questions

Cache dirty bytes rising Are writes outpacing eviction/checkpoint? Did index count or batch size change?
Application threads evicting Is foreground latency tied to cache triggers and dirty-page reconciliation?
Disk full after deletes Is the space reusable internally, pinned by checkpoints, or actually needing compaction?
Aggregation spills Can index order, predicate movement, or cardinality reduction remove blocking memory?
Write conflicts Are transactions or hot documents creating contention at the storage visibility boundary?
Slow restart How much log replay or rollback-to-stable work is needed after the last checkpoint?

Checklist

A practical diagnostic order

1. Confirm operation shape Look at command, namespace, plan summary, keys/docs examined, spills, locks, write concern, and duration. Profiler / explain
2. Check storage pressure Correlate wiredTiger cache, eviction, checkpoint, transaction, and block-manager counters over the same interval. serverStatus rates
3. Check platform pressure Measure disk latency, queueing, CPU, filesystem cache availability, and replication apply behavior. OS / Atlas / cloud metrics

Recap

The engine mental model

Writes

Insert work is collection B-tree mutation plus index B-tree mutation plus transaction visibility plus durability contract.

Reads

Aggregation work is optimizer/planner shape plus storage cursor traversal plus cache/version resolution plus possible stage spill.

Operations

Most symptoms are one of three pressures: too many pages touched, too many pages dirty, or too many versions retained.

Takeaways

How to think in WiredTiger terms

  • Every index is a real B-tree with write, cache, checkpoint, and recovery cost.
  • Every long-running snapshot can preserve old versions and slow cleanup.
  • Every missing or misordered index turns query semantics into extra page traffic.
  • Every durability setting is a contract across server, engine, journal, replication, and storage media.
  • Every tuning change should be checked against operation shape before changing cache or compression settings.
