Operational symptoms map back to specific storage-engine mechanisms

Symptom
Likely engine mechanism
Evidence to inspect
Write latency spikes Inserts wait behind dirty cache pressure, journal sync, index maintenance, or checkpoint I/O.
Dirty eviction or checkpoint pressure Application threads may help evict when eviction cannot keep cache below trigger thresholds.
Metrics wiredTiger.cache, wiredTiger.transaction, opcounters, disk latency, writeConcernError
Reads suddenly scan Planner lost an indexable shape, selectivity changed, or aggregation reordered into a blocking fetch.
Cold pages or wide fetches Index scan touches too many keys, collection fetch pulls many pages, history store may satisfy old snapshots.
Metrics explain keys/docs examined, executionTimeMillis, page read counters, cache misses
Disk grows after deletes Pages are reusable internally, but files may not shrink without compaction and old checkpoints can hold blocks.
Checkpoint and extent reuse Deleted pages become reusable after older checkpoints no longer reference them; truncation only helps at file end.
Metrics collStats, dbStats, compact outcomes, checkpoint progress, filesystem free space
Replication lag under load Primary write path plus oplog/journal pressure increases commit and apply latency.
Storage plus replication coupling Do not tune cache alone if the actual bottleneck is durable write concern or secondary apply.
Metrics rs.printSecondaryReplicationInfo(), repl lag, journal latency, iostat, Atlas metrics
